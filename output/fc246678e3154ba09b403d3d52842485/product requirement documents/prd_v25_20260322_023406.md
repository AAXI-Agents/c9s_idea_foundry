# Executive Summary

**Problem Statement**
Mid-market and enterprise finance teams utilizing NetSuite and QuickBooks Online (QBO) struggle with native system exports that produce highly nested, fundamentally unusable data structures. Consequently, highly paid finance professionals waste countless hours during month-end close performing manual ETL processes—un-nesting JSON, running complex VLOOKUPs, and re-mapping data—just to fit raw ERP output into the rigid spreadsheet templates required by auditors and investors.

**Target Audience**
*   **Primary:** Controllers, FP&A Analysts, and Financial Systems Managers.
*   **Secondary:** External Auditors and Executive Leadership (CFOs) who rely on the timeliness and accuracy of financial reporting.

**Proposed Solution & Key Differentiators**
We are building an AI-driven semantic data extraction and schema mapping engine that transforms raw ERP API data into business-ready formats. Through a React + TypeScript frontend, users simply upload an empty target spreadsheet (CSV/XLSX) and select their desired NetSuite or QBO record type. 

Our backend (Python 3.11 and FastAPI) utilizes crewAI to orchestrate a workflow where Google Vertex AI analyzes the semantic meaning of the uploaded headers and automatically generates the precise transformation logic to map the complex, nested ERP data to the user's custom template. 

*Key Differentiators include:*
*   **Zero Data Mutation Guarantee:** To eliminate the catastrophic risk of AI financial hallucinations, Vertex AI is strictly firewalled from the data payload. The AI *only* writes the schema mapping rules based on metadata; deterministic Python functions execute the actual data movement.
*   **Persistent Mapping:** Approved mappings are saved in MongoDB Atlas, allowing future extractions to run instantly without requiring repeated LLM inference.
*   **Record-Type Specificity:** Built-in semantic understanding of core financial relationships (e.g., how header-level customer data relates to line-item GL accounts).

**Business Impact & Success Criteria**
This feature directly targets operational inefficiency and data confidence during the financial close process. Success for the MVP will be measured by the following SMART criteria:
1.  **Time-to-Value:** Reduce the time required to generate a formatted month-end reconciliation report from an average of 2 hours to under 2 minutes.
2.  **Mapping Accuracy:** Vertex AI must correctly identify and map ≥90% of standard ERP fields to user-defined headers on the first pass, minimizing manual mapping corrections.
3.  **Auditability & Fidelity:** Achieve 100% cryptographic proof that all numerical values extracted from the QBO/NetSuite APIs perfectly match the final XLSX/CSV output without mutation.

**High-Level Phasing**
*   **Phase 1 (MVP):** API integrations for QBO and NetSuite standard records (Invoices, Journal Entries), Vertex AI schema mapping logic, and deterministic Python data extraction.
*   **Phase 2:** React frontend for template upload, human-in-the-loop mapping approval, and MongoDB Atlas mapping persistence. 
*   **Phase 3:** Advanced cryptographic audit logging and general availability release.

# 1. Architecture Overview

The "Intelligent Audit & Close Engine" enforces a strict architectural firewall between non-deterministic AI processes (schema mapping) and deterministic execution (financial data movement). The system is built on a modern, async-first stack designed for high I/O throughput and absolute cryptographic data lineage.

### Technology Stack & Rationale
*   **Backend:** Python 3.11 + FastAPI. *Rationale: Optimal for heavy data manipulation (pandas), JSONPath processing, and native crewAI/Vertex AI integration.*
*   **Frontend:** React + TypeScript. *Rationale: Strict type safety for complex grid UIs and mapping review screens.*
*   **Persistence:** MongoDB Atlas. *Rationale: Document model perfectly fits the highly variable, nested schema configurations of ERP data and few-shot memory arrays.*
*   **AI/Orchestration:** Google Vertex AI + crewAI. *Rationale: Vertex AI provides reliable structured JSON outputs. crewAI orchestrates the schema discovery tasks.*
*   **Background Processing:** Celery + Redis. *Rationale: Required for asynchronous, long-running ERP data extractions and pagination handling without blocking the API.*

# 2. Component Breakdown

## 2.1 API Gateway & Core Services (FastAPI)
*   **Purpose:** Handles REST routing, input validation (Pydantic), auth, and rate limiting.
*   **Dependencies:** `fastapi`, `pydantic`, `motor` (async MongoDB), `pyjwt`.
*   **Key Interfaces:**
    *   `POST /api/v1/templates/upload` (multipart form parser)
    *   `POST /api/v1/mappings/generate` (triggers crewAI task)
    *   `POST /api/v1/jobs/extract` (queues Celery task)

## 2.2 AI Schema Mapping Engine (crewAI + Vertex AI)
*   **Purpose:** Analyzes metadata to write `jsonpath-ng` extraction rules. Strictly isolated from the extraction worker.
*   **Dependencies:** `crewai`, `google-cloud-aiplatform`, `jsonpath-ng`.
*   **State Machine (SchemaMappingRule):**

```ascii
      [Draft]
         |
    (User Trigger)
         v
    [Generating] <-----------------+
         |                         | (jsonpath syntax error)
  (crewAI/Vertex Run)              |
         |                         |
         +--> (Success) --> [PendingReview]
         |                         |
         +--> (Error > 3 retries)  | (User Edits/Approves)
         v                         v
      [Failed]                 [Active]
                                   |
                             (New Mapping Replaces)
                                   v
                             [Deprecated]
```

## 2.3 Extraction Execution Worker (Celery/Pandas)
*   **Purpose:** Deterministic financial data movement. Connects to ERP APIs, paginates records, applies JSONPaths, performs Cartesian un-nesting, and hashes output.
*   **Dependencies:** `celery`, `redis`, `pandas`, `requests`, `jsonpath-ng`.
*   **State Machine (ExtractionJob):**

```ascii
      [Queued]
         |
  (Worker picks job)
         v
      [Running]
         |
  (Fetch & Un-nest)
         |
         +--> (ERP API Error / 5xx) ----> [Failed]
         |
         +--> (Success) ----------------> [Completed] (Sets fileUrl, fileHash)
```

## 2.4 Template Ingestion Engine
*   **Purpose:** Parses CSV/XLSX to extract headers, uses lightweight Vertex AI call to infer schema data types (string, float, etc.).
*   **State Machine (ReportTemplate):**
    `[Draft] ---> (User Confirms) ---> [Active] ---> (User Deletes) ---> [Archived]`

# 5. Error Handling & Failure Modes

| Component | Failure Mode | Classification | Handling Strategy |
| :--- | :--- | :--- | :--- |
| **Vertex AI Profiling** | API Timeout / 503 | Major | Catch exception, fallback to `string` for all columns. Do not block template creation. Log warning. |
| **crewAI Mapping** | Generated invalid JSONPath | Major | Catch via `jsonpath-ng.parse()`. Increment retry loop inside worker. After 3 tries, set status to `Failed` and alert UI. |
| **ERP Extraction** | Rate Limit (HTTP 429) | Minor | Explicit catch for 429s. Implement exponential backoff (1s, 2s, 4s, 8s) up to 5 times before failing job. |
| **ERP Extraction** | Missing Key in payload | Critical | If mapped field is entirely missing from JSON, `jsonpath-ng` returns empty list. Pandas fills with `NaN`/`None`. Do NOT fail job; output empty cells for auditing. |
| **Backend API** | Invalid Template Upload | Minor | Fast fail via Pydantic model validation. Return HTTP 422 with clear validation message array. |

# 7. Security & Trust Boundaries

*   **Architectural Firewall (The Zero Mutation Guarantee):** Live ERP data *never* enters Vertex AI, crewAI, or any LLM memory. The LLM payload consists exclusively of hardcoded schema dictionaries (keys, types) and user-uploaded headers. 
*   **Authentication & Authorization:** 
    *   All endpoints require JWT.
    *   Multi-tenant isolation: `organizationId` is strictly bound to the JWT claims. Every DB query must include `organizationId` to prevent cross-tenant data leakage (IDOR).
*   **Cryptographic Auditability:**
    *   The "Auditor's Receipt" utilizes SHA-256. Once the Excel file is built in RAM by pandas, the byte stream is hashed *before* upload to S3.
*   **Input Validation:**
    *   Template upload size capped at 5MB.
    *   JSONPath injections prevented by passing user input strictly through the `jsonpath-ng.parse()` AST builder; no `eval()` or string execution is ever used.

## Problem Statement

The financial close and audit process relies heavily on the accurate extraction and formatting of ERP data. However, for mid-market and enterprise organizations using systems like NetSuite and QuickBooks Online (QBO), a massive operational gap exists between how financial systems store data and how external stakeholders (auditors, investors, board members) consume it. 

Currently, finance teams operate as manual data translators, bridging this gap through brute-force spreadsheet manipulation or unreliable "5-star" AI tools. This current state introduces severe operational inefficiencies, unacceptably high risk of data mutation, and a complete loss of auditable data lineage.

**The Current State: The Structural, Semantic, and Latency Barriers**
When a Controller or FP&A Analyst needs to provide a Revenue Roll-forward or a Journal Entry reconciliation to an external auditor, they face three distinct barriers:

1.  **The Structural Barrier (Nested Data):** ERPs architect transactional data hierarchically. A single Invoice export contains header-level data (Customer, Date, Total Amount) mixed with variable arrays of line-item details (Item Account, Class, Location, Tax Lines). Standard extraction tools produce heavily nested, normalized tables that are functionally unusable in a standard spreadsheet grid.
2.  **The Semantic Barrier (Brittle Mappings):** External auditors mandate rigid spreadsheet templates (e.g., standard PwC or EY Prepared-By-Client templates). The column headers in these templates rarely match the native API schema of the ERP. For example, an auditor's request for "GL Account Code" must be manually mapped to QBO’s `Line.DetailType.JournalEntryLineDetail.AccountRef.value`. Furthermore, these manual mappings are inherently brittle; they break and require complete rebuilds whenever the ERP updates its API version or the auditor modifies their requested template format.
3.  **The Latency Barrier (The "Static Snapshot" Trap):** Because the translation process is manual and time-consuming, the resulting spreadsheet is a "frozen" snapshot. If a late-arriving adjustment is posted in the ERP (common during month-end), the entire manual translation process must be restarted. This creates a version-control nightmare where auditors may be reviewing data that no longer matches the General Ledger.

**The Business Impact and Pain Points**
Because the tooling fails at the translation layer, highly paid finance professionals are forced into high-risk workarounds. The impact manifests in three critical areas:

*   **The Time Tax & Human Fatigue:** Finance teams spend an average of 2 hours per complex reconciliation report just formatting data. This repetitive, low-value work leads to a high "Boredom-Error" correlation where mistakes are inevitable, and forces teams into "weekend work" to hit reporting deadlines.
*   **The Mutation Risk (Human & Shadow AI):** Every manual spreadsheet manipulation is a vulnerability. A broken formula or an accidental copy-paste error can easily mutate financial figures. Moreover, as desperate teams attempt to use generic, off-the-shelf LLMs to "un-nest" or map this data, they introduce severe **hallucination risk**. An AI that non-deterministically alters a decimal point or sign destroys mathematical certainty. In an audit, these invisible errors lead to restatements and catastrophic loss of institutional trust.
*   **Destruction of Data Lineage & Career Anxiety:** Once raw ERP data is manually transformed in Excel, the connection to the source system is severed. When an auditor points to a specific cell and asks, "Where exactly did this number come from?", the analyst cannot answer without reverse-engineering their own manual steps to find the source NetSuite Transaction ID. This "black box" creates intense professional liability and anxiety, as the Controller cannot prove with 100% certainty that the data hasn't been tampered with or miscalculated.

**Why Now?**
Regulatory scrutiny and external reporting demands are accelerating, requiring faster, more frequent, and more granular financial reporting. Furthermore, the increasing speed of AI-driven business operations renders manual, human-in-the-loop data translation competitively non-viable. The market requires a solution that eliminates the manual "data manipulation" phase entirely, guaranteeing that the numbers exported from the ERP arrive in the exact semantic format requested by the auditor, with absolute mathematical certainty, zero hallucination risk, and perfect cryptographic data lineage.

## Edge Cases

### 1. Structural Mismatches & Cartesian Explosions

**EC-001: Independent Multi-Level Array Un-nesting**
*   **Boundary Condition:** An ERP payload contains multiple independent arrays at the line-item level (e.g., an Invoice containing 5 `LineItems` and 3 `TaxDetails` that are not hierarchically related).
*   **Impact:** A standard Cartesian cross-join would create 15 rows (5x3), duplicating financial totals and breaking mathematical certainty.
*   **System Behaviour:** The Celery execution worker MUST identify independent arrays during the JSONPath phase. It MUST implement a "Sequential Union" expansion:
    1.  Rows 1–5 contain `LineItem` data; all `TaxDetail` columns are explicitly set to `Null`.
    2.  Rows 6–8 contain `TaxDetail` data; all `LineItem` columns are explicitly set to `Null`.
    This ensures that pivoting the final XLSX accurately preserves the ERP Total without row multiplication.

**EC-002: Ragged Hierarchies & Missing Nodes**
*   **Boundary Condition:** ERP records where custom fields exist for some transactions but the keys are entirely absent from the JSON of others.
*   **Impact:** Parser failure or column misalignment.
*   **System Behaviour:** As mandated by FR-015, if a valid JSONPath returns no results, the engine must inject a `NaN`. The Pandas DataFrame construction MUST be initialized using the *Template Headers* as a fixed schema, ensuring consistent column positioning regardless of source node presence.

**EC-003: Circular JSONPath References**
*   **Boundary Condition:** A user manually inputs, or the AI hallucinates, a recursive JSONPath expression (e.g., `$..*`) that forces the parser into an infinite loop on a deeply nested ERP payload.
*   **Impact:** Celery worker hangs indefinitely, leading to queue starvation and process timeout.
*   **System Behaviour:** The `jsonpath-ng` parser MUST be executed with a strict execution timeout (e.g., 5 seconds per record). If the timeout is breached, the job MUST fail gracefully with a `RECURSION_TIMEOUT` error, prompting the user to refine their mapping path.

### 2. Template Parsing & User Input Anomalies

**EC-004: Duplicate Column Headers in Uploaded Template**
*   **Boundary Condition:** User uploads an XLSX with two columns named "Date" (e.g., Invoice Date and Due Date).
*   **Impact:** JSON key collisions during internal processing.
*   **System Behaviour:** The backend must detect duplicate strings. It MUST automatically append an internal programmatic suffix (e.g., `Date_1`, `Date_2`) for the mapping logic while ensuring the final XLSX export output header row preserves the original duplicate strings to satisfy the auditor's template.

**EC-005: Massive Horizontal Templates (>255 Columns)**
*   **Boundary Condition:** Legacy reconciliation templates with hundreds of columns.
*   **Impact:** Vertex AI context window limits or timeout.
*   **System Behaviour:** If the template exceeds 100 columns, the FastAPI backend MUST chunk the inference payload into batches of 50 columns. The crewAI orchestrator will await all batch completions and merge the resulting `jsonpath-ng` arrays before presenting the "Pending Review" status.

### 3. AI Mapping & Integrity Guardrails

**EC-006: Contradictory Few-Shot Memory Overrides**
*   **Boundary Condition:** Two different users in the same organization map the same term to different ERP paths.
*   **Impact:** AI logic "ping-pongs," degrading future inference accuracy.
*   **System Behaviour:** The system MUST implement a "Recency-as-Truth" policy. The `PUT /api/v1/mappings/{id}/approve` endpoint MUST overwrite previous organization-specific mappings with the latest human-approved JSONPath, while maintaining a history log for Sarah (the Risk Owner) to review.

**EC-007: Floating-Point Precision Mutation**
*   **Boundary Condition:** High-volume extractions where thousands of currency line items are summed.
*   **Impact:** Silent data mutation where `float64` rounding errors create a $0.01 variance, destroying "Mathematical Certainty."
*   **System Behaviour:** The deterministic execution engine MUST NOT use `float` types. It MUST cast all numeric fields mapped to "Float" or "Currency" headers to the `decimal.Decimal` type during the Pandas flattening phase to ensure 100% precision.

**EC-008: Currency Pairing Guardrail**
*   **Boundary Condition:** The Power User maps an "Amount" header but neglects to map a "Currency" header for a multi-currency subsidiary.
*   **Impact:** The report may sum amounts in different currencies (e.g., USD and CAD) as a single integer, creating a catastrophic financial error.
*   **System Behaviour:** If crewAI identifies a column as an "Amount" or "Total," it MUST also attempt to map a "Currency Code" node. If no currency field is mapped, the UI MUST display a high-severity warning: *"Warning: You are extracting financial amounts without a corresponding Currency Code. This may lead to inaccurate totals."*

**EC-009: The "Midnight Pivot" Date Shift**
*   **Boundary Condition:** ERP transaction timestamps are stored in UTC. An extraction for a specific date range includes a transaction posted at 11:50 PM EST, which registers as the "next day" in UTC.
*   **Impact:** The exported report shows transactions on the wrong date, triggering a "Reconciliation Variance" and auditor skepticism.
*   **System Behaviour:** The system MUST fetch the 'Company Preference Timezone' from the ERP API during the pre-flight check. The deterministic Python engine MUST localize all UTC timestamps to the ERP’s home timezone before flattening to ensure the 'Date' in the XLSX matches the 'Date' in the ERP UI.

### 4. Extraction Volatility & Infrastructure

**EC-010: ERP Payload Exceeds Worker RAM (OOM Risk)**
*   **Boundary Condition:** A historical extraction resulting in a 3GB+ raw JSON payload.
*   **Impact:** Celery Worker container crash (OOM).
*   **System Behaviour:** The worker MUST NOT load the full JSON blob into RAM. It MUST use the `ijson` streaming parser to iteratively extract records. For payloads > 50,000 records, the worker MUST process in batches of 10,000, writing intermediate results to a persistent disk buffer before final conversion.

**EC-011: Runtime Schema Drift**
*   **Boundary Condition:** An ERP field is modified or deleted by a NetSuite Admin *after* a mapping is approved but *before* job execution.
*   **Impact:** Incomplete or corrupted data extraction.
*   **System Behaviour:** The worker MUST perform a "Pre-flight Schema Check." If a JSONPath in the approved mapping no longer exists in the ERP metadata, the job MUST fail immediately with a `SCHEMA_DRIFT_DETECTED` error, alerting Sarah (the Controller) to re-map the template.

**EC-012: Mid-Stream Auth Token Expiration**
*   **Boundary Condition:** A massive extraction (e.g., 500,000 records) takes longer than the OAuth 2.0 access token's lifespan.
*   **Impact:** Job failure at 90% completion due to a 401 Unauthorized error, wasting compute resources and time.
*   **System Behaviour:** The Celery worker MUST implement proactive token-refresh logic. The worker MUST check the token's expiration timestamp every 10,000 records. If the token is within 5 minutes of expiration, it MUST utilize the Refresh Token to obtain a new session before continuing the stream.

## Error Handling

### 1. AI & Semantic Mapping Failures

**ERR-001: Vertex AI Inference Timeout / Unavailable (HTTP 503/504)**
*   **Trigger:** Google Vertex AI fails to respond within the 15-second SLA during template ingestion or schema mapping.
*   **System Action:** The FastAPI backend MUST catch the exception and prevent the user flow from failing completely. It MUST implement Graceful Degradation by defaulting all un-mapped columns to `String` types and returning a `206 Partial Content` status.
*   **User Experience:** The React UI displays a yellow warning banner: *"AI inference is currently degraded. We couldn't automatically map your columns, but you can proceed by mapping them manually."*

**ERR-002: Invalid JSONPath or Semantic Logic Mismatch**
*   **Trigger:** The crewAI agent generates a `jsonpath-ng` string that fails Python AST compilation OR the system detects a "Semantic Sanity" mismatch (e.g., Vertex AI attempts to map a "Total Amount" template header to an ERP "Date" node).
*   **System Action:** The backend intercepts the error during the pre-execution validation phase. It MUST increment an internal retry counter. If 3 retries fail to produce a valid and logically sane path, the backend returns a `422 Unprocessable Entity`.
*   **User Experience:** The UI stops the "Generating" spinner, flags the specific row in red, and displays: *"The AI mapping for this field is invalid or logically incompatible. Please enter the mapping path manually."*

### 2. ERP Integration & Network Failures

**ERR-003: ERP API Rate Limiting (HTTP 429)**
*   **Trigger:** The Celery extraction worker hits the NetSuite or QBO API rate limit during a massive data pull.
*   **System Action:** The worker MUST explicitly catch the `429 Too Many Requests` error. It MUST NOT fail the job. It triggers an exponential backoff strategy (e.g., 30s, 60s, 120s, up to 5 times) and pauses the specific Celery task.
*   **User Experience:** The "Jobs" dashboard updates the job status from `Running` to `Throttled - Auto-Retrying` so the user knows the system has not crashed.

**ERR-004: OAuth Token Expiration & Atomic Integrity (HTTP 401)**
*   **Trigger:** During a long-running extraction, the QBO/NetSuite access token expires mid-stream.
*   **System Action:** The Celery worker MUST pause the extraction and attempt to use the Refresh Token to acquire a new session. If the refresh is successful, it resumes from the last pagination cursor. If the refresh fails, the system MUST enforce Atomic Integrity by issuing a `DELETE` command to the S3 staging path for that `job_id`, ensuring no partial byte-stream is ever hashed or finalized. The job status is set to `Failed`.
*   **User Experience:** If the refresh fails, the UI notifies the user: *"Your ERP connection expired during extraction. Please re-authenticate and restart the job."* Users are never shown incomplete or "partial" financial reports.

### 3. Execution & Transformation Failures

**ERR-005: Infrastructure Limits (Disk/Time Exhaustion)**
*   **Trigger:** A requested date range returns a payload that exhausts the disk-based chunking capacity (90% utilization) or exceeds the maximum 2-hour execution safety window.
*   **System Action:** The worker detects the threshold breach. It MUST halt the job, purge any intermediate disk chunks to free space, and update the MongoDB status to `Failed`. 
*   **User Experience:** Returns a `413 Payload Too Large` to the frontend. The UI displays: *"This request exceeds the maximum system capacity. Please reduce your date range or contact support for an enterprise-tier historical extraction."*

**ERR-006: Runtime Schema Drift & Transformation Type Mismatch**
*   **Trigger:** The deterministic Python engine attempts to execute an approved JSONPath, but the underlying custom field was deleted in the ERP by an Admin (Schema Drift), OR the ERP returns data that cannot be safely cast into the template’s required type (e.g., text found in a `decimal.Decimal` currency column).
*   **System Action:** To guarantee "Zero Data Mutation," the job MUST halt immediately upon the first un-castable record. The system MUST NOT finalize the file. Partial S3 data is deleted, and the status is set to `Failed`.
*   **User Experience:** The Job status changes to `Failed`. The UI explicitly states: *"Extraction halted. The ERP data for field [Field Name] at Record ID [ID] is missing or incompatible with your template. Please update your mapping to continue."*

### 4. User Input & Validation Failures

**ERR-007: Invalid Template Upload (Format/Size)**
*   **Trigger:** A user attempts to upload a `.pdf`, a corrupted `.csv`, or an `.xlsx` file exceeding the 5MB limit.
*   **System Action:** The FastAPI gateway intercepts the request before parsing. Returns an immediate `400 Bad Request` (format) or `413 Payload Too Large` (size).
*   **User Experience:** The drag-and-drop zone outlines in red and states: *"Invalid file. Please upload an empty .CSV or .XLSX file under 5MB."*

**ERR-008: Conflicting Few-Shot Manual Overrides**
*   **Trigger:** A user tries to map a single ERP field to two different target template columns, creating a Cartesian conflict.
*   **System Action:** The `PUT /api/v1/mappings/{id}/approve` endpoint validates the mapping array. It detects duplicate ERP source paths and rejects the save with a `409 Conflict`.
*   **User Experience:** The UI prevents the "Save Mapping" action, highlighting the conflicting rows: *"You cannot map the same ERP field to multiple columns. Please resolve the conflict."*

### 5. Logging and Alerting Strategy

*   **Error Taxonomy:** All backend errors MUST log to stdout in JSON format using standard Python `logging`.
*   **Contextual Payloads:** Every error log MUST include `trace_id`, `organization_id`, and `user_id`.
*   **Data Masking:** Error logs MUST NEVER contain financial payloads, PII, or raw JSON snippets from the ERP. Exceptions must be caught and scrubbed before logging.
*   **Critical Alerts:** PagerDuty/Datadog alerts MUST trigger for the DevOps team if:
    *   The `429 (Rate Limit)` failure rate exceeds 5% in a 10-minute window.
    *   Any `5xx` errors originate from the Celery extraction workers.
    *   The SHA-256 integrity check fails for a finalized job.

*   **Vertex AI Structured Output & Regionality:** We assume that Google Vertex AI will consistently adhere to the requested structured JSON formatting for `jsonpath-ng` generation. Furthermore, we assume that Vertex AI and the specific LLM models used by the crewAI orchestration layer are fully available in the specific cloud regions (e.g., `europe-west3`) required to satisfy our clients' Data Sovereignty compliance (NFR-004).
*   **Source Data Integrity:** We assume that transactional data within the ERP is structurally consistent enough for deterministic parsing. Specifically, we assume that fields mapped as "Numeric" or "Date" do not contain unparseable, unstructured text (e.g., "See notes") that would cause the Pandas execution engine to halt with a Type Mismatch error.
*   **Infrastructure Elasticity:** We assume that the underlying cloud infrastructure (managed via DevOps) can automatically scale Celery worker nodes to meet Month-End Close concurrency limits (e.g., 100 concurrent jobs of 10,000+ records) without triggering OOM (Out-of-Memory) cascading failures or breaching the 60-second queue wait time (NFR-006).

### 2. User & Workflow Assumptions

*   **Templating Rigidity:** We assume that auditors and FP&A teams have established reporting requirements. The product assumes users will upload a well-defined `.csv` or `.xlsx` template containing explicit column headers representing their final desired state. If users expect the system to magically invent a reconciliation format from scratch, the Time-to-Value metric (SM-001) will fail.
*   **Human-in-the-Loop Willingness:** We assume that the "Power User" (David persona) is willing to invest up to 60 seconds reviewing and approving the AI-generated mappings. The "Zero Data Mutation Guarantee" relies entirely on the premise that a human validates the semantic mapping logic *before* the deterministic Python extraction begins.
*   **Audit Portal & Portable Lineage:** We assume that external auditors (Elena persona) will be willing to log into our Secure Auditor Portal to verify cryptographic hashes. For highly restrictive audit firms that prohibit the use of external third-party portals, we assume the cryptographic "Auditor's Receipt" embedded directly in the generated XLSX file (Sheet 2) is legally and operationally sufficient to prove data lineage.

### 3. Business & Market Assumptions

*   **Trust Trumps 'Fully Autonomous' AI:** We assume our core market hypothesis is true: Enterprise CFOs and Controllers prioritize 100% mathematical certainty and auditability over "hands-free, fully automated" LLM pipelines. Our architectural firewall is our primary competitive advantage; if the market prefers riskier, faster 100% automated systems, our approach will be a competitive disadvantage.
*   **Customer Willingness to Pay:** We assume that reducing month-end reporting time from 2 hours to 2 minutes—and providing mathematical certainty—represents a high enough ROI that mid-market and enterprise organizations are willing to pay standard B2B SaaS Enterprise tier pricing for the solution.
*   **Legal/Security Acceptance:** We assume that the CISO and internal security teams will accept the "Architectural Firewall" as sufficient mitigation against data privacy risks. We assume they will agree that passing *schema metadata* (keys, types) to Vertex AI does not violate PII/PCI/GDPR constraints, as no actual financial payload is transmitted to the LLM.
