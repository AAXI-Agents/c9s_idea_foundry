# Executive Summary

**Problem Statement**
Migrating high-volume, unstructured client financial data into systems like QuickBooks Online (QBO) currently relies on brittle, manual mapping templates that break instantly upon encountering API limits or missing ledger codes. When legacy uploaders fail, they present cryptic API errors that force accounting teams to manually halt workflows, decipher spreadsheet anomalies, and re-attempt uploads—a severe operational bottleneck that threatens ledger integrity and drains billable hours.

**Target Audience**
* **Client Advisory Services (CAS) Firms:** Accounting teams managing high-volume transactional data (e.g., e-commerce sales, payroll runs) across multiple client QBO environments.
* **Accounting Systems Integrators:** Technical consultants tasked with migrating historical financial data from legacy on-premise software into modern cloud ERPs.

**Proposed Solution**
The AI-Powered Accounting Data Orchestration Engine is an intelligent middleware platform that transforms raw financial data ingestion into a frictionless, automated workflow. Built on a modern React and TypeScript frontend, the system abstracts the complexity of asynchronous data processing. At its core, it leverages Google Vertex AI for Intelligent Semantic Ingestion—automatically mapping unstructured CSV/Excel columns to strict QBO schemas without manual templates. When Intuit's APIs reject a payload, a Python/FastAPI backend deploying crewAI micro-agents intercepts the error, interprets the accounting context, and autonomously resolves the validation issue in real-time. 

**Key Differentiators**
* **Agentic Error Resolution:** Unlike "dumb pipe" legacy competitors (e.g., SaasAnt), the platform uses crewAI to auto-correct unmapped sub-customers, inactive class codes, and rate-limit rejections without user intervention.
* **Zero-Setup Ingestion:** Google Vertex AI eliminates the need for rigid column-mapping drop-downs, semantically understanding context (e.g., mapping "Amex Charges" to `JournalEntry` lines).
* **Guaranteed Auditability:** Utilizing MongoDB Atlas, the system maintains immutable logs of every ingested file and API payload, enabling a one-click rollback feature that meticulously reverses API calls to restore pristine ledger states.
* **Agnostic Scalability:** The core Python/crewAI engine is architected abstractly, providing a scalable foundation ready to expand from QBO to enterprise endpoints like the NetSuite REST API.

**Business Impact and Success Criteria**
The platform will enable CAS firms to scale their client base aggressively without a proportional increase in administrative headcount. 
* **Time-to-Value:** Reduce initial client data onboarding and file-mapping time by 80% compared to legacy CSV templates.
* **Automation Efficacy:** Achieve a >90% auto-resolution rate for standard QBO API validation and rate-limit errors via crewAI agents.
* **Data Integrity:** Ensure 0% permanent ledger corruption with a 100% success rate on one-click rollbacks for duplicate or erroneous batch uploads.
* **Performance:** Maintain 99.9% uptime with API concurrency handling that completely eliminates QBO `HTTP 429 Too Many Requests` penalties.

**High-Level Timeline**
* **Phase 1 (Months 1–2): Semantic Ingestion & Core Wedge.** Develop React/TypeScript frontend, Vertex AI semantic mapping, and foundational Python/FastAPI backend for QBO integration.
* **Phase 2 (Months 3–4): Agentic Pipeline & Auditability.** Deploy crewAI error-resolution agents and implement MongoDB Atlas payload logging with one-click rollback capabilities.
* **Phase 3 (Months 5–6+): Enterprise Expansion.** Generalise the orchestration engine to support complex enterprise endpoints, beginning with the NetSuite REST API.

### 1.2 Technology Stack Rationale
*   **Backend (Python 3.11 + FastAPI):** Python is mandatory for crewAI integration. FastAPI provides native async/await necessary for high-throughput I/O bound tasks (API syncing) without blocking the GIL.
*   **Database (MongoDB Atlas):** Financial payloads vary drastically between ERP entities. MongoDB's flexible schema handles unstructured API payloads (Mixed JSON) naturally, while allowing strict schema validation at the application layer via Pydantic.
*   **AI (Google Vertex AI & crewAI):** Vertex AI provides the LLM reasoning backbone. crewAI orchestrates multi-agent workflows (Diagnostic Agent -> Resolution Agent) for multi-step ERP error resolution.
*   **Frontend (React + TypeScript):** Enforces strict typing for the complex mapping and nested JSON resolution payloads presented to the user.

### 1.3 Data Flow Architecture

```text
UPLOAD FLOW (Happy & Empty/Nil Paths)
[CSV File] -> (FastAPI) -> Has Rows? 
    ├── NO (Empty Path): -> Return 400 Bad Request
    └── YES (Happy Path): -> Chunk File -> Send Sample to Vertex AI
           -> Vertex Returns JSON Mapping -> Save `SchemaMapping` to MongoDB
           -> WebSocket Emit -> UI Shows "Confidence Pulse" -> User Approves
           -> Enqueue `SyncTask`s.

SYNC & ERROR FLOW (Error & Resolution Paths)
[Queue] -> Pop `SyncTask` -> Send Payload to QBO API
    ├── 200 OK: -> Save `AuditLogEvent` -> Mark `SyncTask` Success
    ├── 429 Rate Limit: -> Backoff -> Requeue
    └── 4xx Validation Error: 
           -> Route to crewAI (Diagnostic + Resolution Agents)
           -> crewAI Contextualizes Error -> Can Auto-Resolve?
                 ├── YES (High Confidence): Generate Fix -> Requeue Task
                 └── NO (Low Confidence): Generate `RequiresIntervention` Alert
                        -> Save `SyncError` -> Await User Input.
```

## 2. Component Breakdown & State Machines

### 2.1 Component A: Intelligent Semantic Ingestion Pipeline
*   **Purpose:** Translate raw, unmapped CSV data into normalized JSON payloads based on ERP schemas.
*   **Dependencies:** Google Vertex AI (gemini-1.5-pro), MongoDB (`IngestionBatch`, `SchemaMapping`).
*   **Failure Modes:** Malformed CSV, Vertex AI timeout, LLM Hallucination (returns invalid JSON).
*   **State Machine (`IngestionBatch`):**

```text
[Pending] -----> [Analyzing] -----> [MappingSuggested] -----> [MappingApproved]
                     |                      |                         |
                     v                      | (User Overrides)        v
                  [Failed] <----------------+                    [Processing]
            (LLM Failure / Bad File)                                  |
                                                                      v
                                                                 (To Sync Queue)
```

### 2.2 Component B: Agentic Data Orchestration (crewAI + FastAPI Workers)
*   **Purpose:** Process the queue deterministically. Handle Intuit API constraints aggressively. Invoke agents *only* on domain-level failures.
*   **Dependencies:** Intuit QBO REST API, crewAI, MongoDB (`SyncTask`, `SyncError`).
*   **Interfaces:** FastAPI background workers using a Leaky Bucket algorithm to enforce strictly < 500 requests / minute per tenant.
*   **State Machine (`SyncTask`):**

```text
                +-------------------+
                |                   v
[Queued] ---> [Syncing] --------> [Failed] (4xx/5xx)
   ^            |                   |
   |            | (200 OK)          v
   |            v           [AgentResolving] --------> [RequiresIntervention]
   |        [Success]               |                          |
   |                                | (AI Auto-Fix)            | (User Fix)
   +--------------------------------+                          |
   |                                                           |
   +-----------------------------------------------------------+
```

### 2.3 Component C: Immutable Audit & Rollback Engine
*   **Purpose:** Provide absolute ledger safety. Every successful 2xx request generates an inverse compensation action.
*   **Dependencies:** MongoDB (`AuditLogEvent`, `RollbackJob`), Vertex AI (Impact Summarization).
*   **Interfaces:** `POST /api/v1/audit/rollbacks/{job_id}/execute`.
*   **State Machine (`RollbackJob`):**

```text
[Draft] ---> [ImpactAssessed] ---> [Reverting] ---> [Success]
                                        |
                                        +---------> [PartialSuccess]
                                        |
                                        +---------> [Failed]
```

## 4. Data Model & Persistence Strategy

### 4.1 Schema Definitions (MongoDB Atlas)
Collections are strictly partitioned by `tenantId` (indexed).

*   `tenants`: Stores QBO OAuth credentials (Encrypted).
*   `ingestion_batches`: Tracks file upload state.
*   `schema_mappings`: Stores Vertex AI map rules.
*   `sync_tasks`: Individual row processing state.
*   `sync_errors`: CrewAI resolution data.
*   `audit_log_events`: Immutable API logs.
*   `rollback_jobs`: Ledger reversion state.

### 4.2 Indexing Strategy
*   **High Write/Read:** `sync_tasks` compound index on `{ batchId: 1, status: 1 }` for fast queue polling.
*   **Uniqueness:** `sync_errors.taskId` (Unique) — 1:1 relationship to prevent duplicate resolutions.
*   **Retrieval:** `audit_log_events` compound index `{ batchId: 1, isRolledBack: 1 }` to rapidly calculate rollback scope.
*   **Data Lifecycle:** TTL index on `audit_log_events.timestamp` set to 7 years for compliance retention.

## 5. Error Handling & Failure Modes

| Component / Action | Failure Mode | Mitigation / Strategy | Severity |
| :--- | :--- | :--- | :--- |
| **QBO API Call** | `429 Too Many Requests` | Caught before AI. Exponential backoff queue (Base: 2s, Max: 60s). | Minor |
| **QBO API Call** | `400 ValidationFault` | Sent to crewAI. If AI fails, state `RequiresIntervention`. | Major |
| **Vertex AI (Ingest)** | Timeout or 5xx | Fallback to empty schema. UI prompts user for Manual Mapping. | Major |
| **crewAI (Resolution)** | Hallucinated Payload | Strict Pydantic validation before QBO dispatch. Fails -> `ManualFix`. | Critical |
| **Rollback Execution** | Record Already Reconciled | Catch Intuit Error. Mark task `Failed`, Job -> `PartialSuccess`. Alert Admin. | Major |
| **WebSockets** | Disconnect during Sync | Frontend falls back to HTTP polling (every 5s) if WS drops. | Minor |

## 6. Test Strategy

We adopt a diamond test profile: heavy on Integration and specific AI determinism tests.

### 6.1 Test Pyramid
*   **Unit Tests (Pytest, 40%):** Strict validation of Pydantic models, JSON mapping transformations, and state machine transition guards.
*   **Integration Tests (Pytest + VCR.py, 40%):** Mocking QBO API responses (happy, 400 validation faults, 429 rate limits) to verify queue behavior and rollback payload generation.
*   **AI Evaluation Tests (15%):** "Golden Dataset" of 50 common QBO errors. CrewAI must output the correct `humanReadableResolution` and `resolutionPayload` with >95% accuracy in CI/CD before deploy.
*   **E2E Tests (Cypress/Playwright, 5%):** Happy path upload, map approve, sync, and full batch rollback against a QBO Sandbox environment.

### 6.2 Critical Paths
*   **Idempotency of Rollbacks:** A rollback executed twice must result in a single set of DELETEs.
*   **Tenant Isolation:** Tests asserting a user token for Tenant A cannot access `IngestionBatch` of Tenant B.

## 7. Security & Trust Boundaries

### 7.1 Attack Surface Analysis
*   **Malicious CSV Payloads:** Prevent CSV injection (Formula injection) by sanitizing inputs before parsing to JSON and before sending to LLM (prevents prompt injection).
*   **API Abuse:** `/api/v1/ingestion/upload` strictly rate-limited (20/hour) and restricted to 50MB file size to prevent memory exhaustion and Vertex AI billing spikes.
*   **Privilege Escalation:** `/rollback` endpoints enforce Admin RBAC middleware.

### 7.2 Data Classification & Encryption
*   **OAuth Credentials:** Encrypted at rest using Application-Level Encryption (AES-256-GCM) with AWS KMS / GCP KMS before being written to MongoDB.
*   **PII/Financial Data:** Stored in `sync_tasks.qboPayload`. MongoDB Atlas encryption at rest (Volume level) is sufficient, but logs emitted to Datadog/CloudWatch MUST strip/mask `qboPayload` data.

## Problem Statement

The core asset of any Accounting Systems Integrator or Client Advisory Services (CAS) firm is the absolute integrity of its financial truth. However, the process of migrating high-volume, unstructured client data—such as e-commerce sales, payroll runs, or legacy software records—into strict destination ERPs like QuickBooks Online (QBO) and NetSuite is currently a brittle, high-risk operational bottleneck. 

**The Failure of "Dumb Pipe" Tooling**
Current market solutions (e.g., SaasAnt, Transaction Pro Importer) function as rigid, stateless data pipes. They mandate that accounting teams manually manipulate messy, real-world source data into highly specific column-mapping templates prior to ingestion. This approach is fundamentally flawed for modern financial operations due to three severe technical limitations:

*   **Context-Blind Ingestion:** Legacy tools possess zero semantic understanding of financial intent and cannot navigate Cross-ERP Schema complexity. They cannot translate unstructured data into domain-specific logic. For example, a human operator must manually deduce that a "Refund" in a CSV requires a `CreditMemo` in QBO, but dictates an entirely different Custom Segment and Internal ID structure in NetSuite. This schema fragmentation creates a massive, non-billable "Mapping Tax" on every new client.
*   **Stateless Operations:** Legacy pipelines have no "memory" or local cache of the destination ERP's configuration. Because they do not maintain a synchronized shadow state of the Chart of Accounts (COA), active Vendors, or custom tracking classes, they are incapable of pre-validating data. They effectively "fly blind," relying entirely on the ERP API to catch errors only after a data push is initiated.
*   **Zero Graceful Error Handling:** Because they lack state awareness, when an ERP API predictably rejects a payload, legacy pipelines instantly break. These failures typically stem from strict API concurrency constraints (e.g., Intuit's `HTTP 429 Too Many Requests`) or domain-level validation rejections (e.g., a `400 ValidationFault` caused by an archived vendor or an unmapped sub-customer). 

**The Forensic Data Nightmare**
When these inevitable rejections occur, the user experience is disastrous. Because legacy tools lack transactional atomicity (the ability to treat a batch as a single unit with safe rollbacks), a single error in row 4,032 of a 10,000-row upload creates a dangerous "Partial Success" state. 

This forces the accountant into a crippling Time-to-Remediation (TTR) loop:
1. **Halt and Audit:** Manually identify which rows were accepted and which were rejected to avoid duplicate entries.
2. **Technical Deciphering:** Translate cryptic API faults (e.g., "Object Not Found") into actionable accounting fixes (e.g., "Reactivate Vendor 'X'").
3. **Manual Mutation:** Edit the source CSVs or log directly into the ERP to create the missing entities.
4. **Data Lock & Re-Attempt Risk:** The accountant suffers from "Data Lock"—a paralyzing fear of re-running the batch, knowing that a misstep could duplicate "successful" rows and irreversibly pollute the ledger.

**Business Impact: "Ledger Anxiety" and the Scalability Ceiling**
This severely degraded workflow generates what firms describe as "Ledger Anxiety." Onboarding a new client feels like performing open-heart surgery in the dark. A single unhandled duplicate batch destroys professional credibility and mandates hours of forensic data janitorial work. 

Ultimately, the business consequence is the "Headcount Trap." Under the current paradigm, scaling a firm's revenue by onboarding new, high-volume clients demands a proportional and unsustainable increase in administrative headcount simply to manage data friction and manual risk mitigation. Firms are artificially capped in their growth because their existing tools fundamentally fail to decouple data ingestion volume from human oversight.

## Edge Cases

### 1. AI & Semantic Mapping Boundaries

**EC-001: Extreme Schema Hallucination (Vertex AI)**
*   **Scenario:** The data sample provided to Vertex AI is highly ambiguous (e.g., headers named `Field1`, `Field2` with purely numerical data). Vertex AI hallucinates a mapping to an inappropriate destination ERP endpoint (e.g., mapping arbitrary numbers to `TaxRate`).
*   **System Behavior / Mitigation:** 
    *   The backend system SHALL strictly validate the LLM's proposed output against the MongoDB Shadow State (the cached version of the ERP's active configuration).
    *   If the AI's confidence score drops below 0.30, or if the proposed fields fail structural schema validation against the ERP metadata, the system must abort the auto-mapping.
    *   The UI must bypass the "Confidence Pulse" entirely, display a "Low Context Alert," and force the user into the manual drop-down mapping flow.

**EC-002: Infinite Agentic Mutation Loop**
*   **Scenario:** The crewAI Resolution Agent intercepts a `400 ValidationFault` (e.g., "Invalid Class Code"). The agent mutates the payload to fix the class, but the mutation inadvertently triggers a new validation rule (e.g., "Class Code requires a Sub-Customer"). The new push fails, returning a new error to the agent, potentially creating an infinite loop of API failures that exhausts rate limits and Vertex AI tokens.
*   **System Behavior / Mitigation:**
    *   The `SyncTask` state machine SHALL strictly enforce the `MaxAgentAttempts` constraint defined in the Functional Requirements (capped at 2 retries).
    *   Upon the second failure of a single `taskId`, the system must immediately sever the agent loop, lock the task, and transition it to the `RequiresIntervention` state for Human-in-the-Loop resolution.

### 2. State & Concurrency Anomalies

**EC-003: "Phantom Success" on Network Timeout**
*   **Scenario:** The FastAPI background worker dispatches a `SyncTask` payload to the NetSuite/QBO API. The ERP successfully processes the payload and commits it to the ledger, but the HTTP connection drops (e.g., `504 Gateway Timeout`) before the worker receives the `200 OK` acknowledgment. 
*   **System Behavior / Mitigation:**
    *   The worker SHALL NOT automatically requeue the task for a blind retry.
    *   Before executing any retry logic, the worker must execute a synchronous `GET` query against the ERP API using a deterministic idempotency key (a hash of the raw CSV row, stored previously in the MongoDB `SyncTask` record) to check if the entity already exists.
    *   If the record is found, the system retroactively logs the `AuditLogEvent` and marks the task as `Success`.

**EC-004: Mid-Batch State Invalidation (ERP Shadow State)**
*   **Scenario:** An `IngestionBatch` of 100,000 rows is currently processing. At row 50,000, an external user logs directly into the QuickBooks Online UI and archives the "Amazon" Vendor or closes the current accounting period. Subsequent rows in the queue relying on that state will suddenly begin to fail en masse.
*   **System Behavior / Mitigation:**
    *   The worker queue will encounter a `400 ValidationFault` for row 50,001.
    *   The crewAI Diagnostic Agent will intercept the error, query the live ERP state, and detect the sudden state mismatch (e.g., "Vendor Archived").
    *   Upon detecting an environmental state change, the system SHALL automatically pause the remainder of the `IngestionBatch` queue to prevent cascading failures. It must trigger a `RequiresIntervention` alert for the CAS Manager to resolve the core state issue before unpausing.

**EC-007: Mid-Ingestion Credential Invalidation (OAuth Decay)**
*   **Scenario:** A 100,000-row batch takes over 3 hours to process via the Leaky Bucket queue. During this window, the QBO OAuth access token expires (typically a 60-minute TTL) and the background refresh-token flow unexpectedly fails (e.g., Intuit API outage or revoked permissions).
*   **System Behavior / Mitigation:**
    *   The system SHALL implement "Pause-on-401" logic.
    *   If a background worker encounters an expired token and cannot refresh it, the worker must immediately transition the entire batch to the `Auth_Suspended` state, securely preserving the current queue position.
    *   The system must dispatch an alert to the Systems Admin (David) to re-authenticate. Upon successful re-authentication, the queue resumes exactly from the last successfully acknowledged row.

### 3. Rollback Integrity Boundaries

**EC-005: Reconciled or Period-Locked Rollbacks**
*   **Scenario:** A user triggers the "Physical Undo Switch" to execute a batch rollback. However, some of the transactions in that batch were ingested weeks ago and have since been cleared in a bank reconciliation, or the month-end books have been formally closed in the ERP.
*   **System Behavior / Mitigation:**
    *   The Rollback Executor Worker SHALL execute a pre-validation check by querying the ERP for the lock status of every transaction *before* issuing compensating `DELETE` or `VOID` commands.
    *   Any transaction flagged as "Reconciled" or residing in a "Closed Period" must be explicitly skipped to prevent ledger corruption.
    *   The system must generate a "Partial Rollback Report" detailing the exact `taskIds` that could not be reverted, providing the CAS Manager with instructions for manual, compliant journal entries.

**EC-006: Interleaved Entity Dependencies**
*   **Scenario:** During ingestion, a crewAI agent autonomously creates a missing parent entity (e.g., a new Customer: "Acme Corp") to satisfy a transaction payload (e.g., an Invoice for Acme Corp). When the user rolls back the batch, deleting the Invoice is straightforward, but it leaves an orphaned, autonomously created Customer.
*   **System Behavior / Mitigation:**
    *   The system SHALL first void or delete the child transaction (the Invoice).
    *   It must then check the ERP (via a `GET` request) to determine if the parent Customer ("Acme Corp") is linked to any other transactions outside the current batch boundary.
    *   If zero dependencies exist, the system SHALL execute a "Soft Delete" or "Archive" command on the parent entity. This returns the ERP to its pristine operational state without destroying the ERP's internal audit ID history.

**EC-008: Simultaneous Sync and Rollback (Batch Locking)**
*   **Scenario:** Sarah triggers a Rollback via the "Physical Undo Switch" while background workers are still actively attempting to push crewAI-resolved "Fixed" payloads for that exact same batch.
*   **System Behavior / Mitigation:**
    *   The system SHALL enforce a strict Mutex State Lock on the `IngestionBatch` record.
    *   If a Rollback is initiated, the system must immediately pause all active `SyncTasks` for that batch ID and successfully cancel them before the rollback executor is permitted to begin issuing `DELETE/VOID` commands.
    *   If tasks are mid-flight and cannot be cancelled, the Rollback request must be temporarily rejected with a "Workers Active" warning, preventing race conditions and duplicate mutations.

## Error Handling

### 1. Error Taxonomy & Strategy

The orchestration engine classifies errors into four distinct categories. Unlike legacy tools that silently fail or abruptly crash the ingestion batch, this system treats errors as manageable states within a continuous workflow. It leverages AI resolution where deterministic and safe, while gracefully failing over to Human-in-the-Loop (HITL) intervention when strict safety constraints are breached.

| Category | HTTP / System Code | Description | Primary Mitigation |
| :--- | :--- | :--- | :--- |
| **Logic / Domain** | `400 ValidationFault` | ERP rejects payload due to accounting rules (e.g., archived vendor, inactive class). | crewAI Agent Interception (Max 2 retries). |
| **System / Network** | `429`, `504`, `5xx` | ERP or AI Provider rate limits breached; connection lost. | Leaky Bucket Queueing & Exponential Backoff. |
| **Cognitive (AI)** | Vertex AI `< 0.85` | LLM cannot determine a safe semantic mapping. | Graceful degradation to Manual UI mapping. |
| **User Input** | `422 Unprocessable` | Malformed CSV (e.g., no headers, corrupted binary). | Synchronous API rejection with clear UI instructions. |

---

### 2. Domain Validation Errors (ERP `400` Responses)

When the destination ERP (e.g., QBO or NetSuite) rejects a payload due to core accounting business logic, the system intercepts the failure to protect the user from cryptic JSON.

*   **Interception & Contextualization:** 
    *   The FastAPI worker catches the `400 ValidationFault`. 
    *   The `SyncTask` state shifts to `AgentResolving`. 
*   **Autonomous Resolution & Auditability:** 
    *   The crewAI Diagnostic Agent queries the ERP for context to formulate a fix.
    *   **Circuit Breaker:** The system SHALL enforce a strict `MaxAgentAttempts=2` limit. If the second autonomous resolution attempt results in another `400 ValidationFault`, the system SHALL immediately transition the task to `RequiresIntervention` to prevent infinite AI logic loops and API token exhaustion.
    *   **Immutable Mutation Diff:** When an agent successfully mutates a payload, the system SHALL persistently log a JSON `diff` comparing the user's raw source data to the agent's mutated payload.
*   **Human-in-the-Loop (HITL) Recovery UX:** 
    *   When a task enters `RequiresIntervention`, the UI SHALL display a deterministic, plain-English translation of the error using a pre-defined mapping template, stripping technical jargon (e.g., displaying *"Item 'Shopify_Sales' does not exist in your Chart of Accounts"*).
    *   Rows successfully altered by agents must display a "Modified by AI" badge. The UI SHALL allow the CAS Manager (Sarah) to review the `diff` and either **Accept**, **Edit**, or **Reject & Re-map**.
    *   **Correction-to-Sync Path:** The UI SHALL provide a "Save & Re-Sync" action. Upon user correction, the task SHALL immediately transition back to the `Queued` state for dispatch, and its internal `AgentAttemptCount` must be reset to zero.

### 3. System and Network Errors (ERP `429` / `5xx` Responses)

System errors are managed strictly by the backend architecture to guarantee transactional idempotency and prevent "Ledger Anxiety."

*   **Throttling & Limits (`429 Too Many Requests`):** 
    *   *System Action:* The worker catching a `429` from either the ERP or the Vertex AI provider SHALL immediately pause the specific tenant thread. It must apply an exponential backoff strategy (Base: 2s, Max: 60s) before requeuing the task. 
    *   *User UX:* The UI abstracts the API pause, displaying "Processing - ERP Throttling Active."
*   **Network Timeouts (`504 Gateway Timeout`):** 
    *   *System Action:* The worker must assume the payload *may* have been processed by the ERP before the connection dropped.
    *   *Idempotency Check:* Before executing any retry, the worker SHALL execute a synchronous `GET` request against the ERP using a deterministic idempotency hash (generated via `SHA-256(Raw_Row_String + Batch_ID + Row_Index)`). If the record exists in the ERP, the system logs the audit event and marks the task as `Success`.

---

### 4. Cognitive Errors (AI Hallucinations / Outages)

Because the system relies on probabilistic modeling, it implements hard constraints to protect the ledger from "hallucinated" data.

*   **Low Confidence Mapping:**
    *   *Trigger:* Vertex AI returns a `SchemaMapping` with a confidence score `< 0.85`, or proposes a field that does not exist in the MongoDB Shadow State (Chart of Accounts).
    *   *System Action:* The system SHALL automatically discard the AI's suggestion for that column.
    *   *User UX:* The UI highlights the column in Red ("Confidence Pulse") and requests manual intervention via a standard drop-down mapping tool.
*   **Intelligence Outage:**
    *   *Trigger:* Google Vertex AI or crewAI endpoints are unreachable.
    *   *System Action:* If the endpoints remain unreachable after 3 exponential backoff attempts, the system SHALL gracefully degrade to a deterministic pipeline.
    *   *User UX:* The UI displays: *"Our AI mapping assistant is currently offline. Please proceed with manual column mapping."*

---

### 5. Dead-Letter Queues (DLQ) & Batch Circuit Breakers

To ensure absolute auditability and prevent catastrophic batch failures, no error is allowed to fail silently.

*   **Max System Retries:** If a `SyncTask` fails three times consecutively for a system/network error (`5xx`), it is moved to a Dead-Letter Queue (DLQ) for engineering review.
*   **Batch-Level Panic Threshold:** If the cumulative error rate for an active `IngestionBatch` exceeds 10% (evaluated strictly after the first 500 rows are processed), the system SHALL immediately **Stop the Dispatcher** for that batch. It must freeze the queue and trigger a high-priority alert to the Technical Integrator (David) to prevent further ledger pollution.
*   **Error Logging Mandate:** Every error record written to the `sync_errors` MongoDB collection MUST contain: the `taskId`, the `raw_api_response`, the `agent_reasoning_log`, and the `mutation_diff` (if an agent altered the payload).

*   **Ledger Accuracy (Lagging Indicator)**

These metrics ensure the FastAPI orchestration layer securely handles high-volume concurrency without triggering ERP penalties or leaking sensitive data.

*   **Trust Boundary Integrity (PII Safety)**

    *   **Measurement:** 
        *   *Interceptor Efficacy:* A 0.1% randomized audit of outbound Vertex AI payloads to verify the PII Anonymization Layer successfully masked sensitive data *before* it left the Trust Boundary.

    *   **Why:** A single PII leak permanently breaches the Trust Boundary, violating enterprise compliance and destroying firm credibility.
*   **ERP Rate-Limit Breaches (Leaky Bucket Efficacy)**
    *   **Target:** 0 occurrences of destination ERP `HTTP 429 Too Many Requests` errors surfaced to the user.
    *   **Measurement:** Absolute count of `429` status codes logged in standard Datadog telemetry.
    *   **Why:** Validates that the internal FastAPI Leaky Bucket dispatcher is perfectly regulating queue concurrency according to strict Intuit/NetSuite tenant limits.
*   **System Uptime SLA**
    *   **Target:** 99.9% uptime.
    *   **Measurement:** Synthetics pinging the `/health` endpoint for the API gateway and background worker services.
*   **Dead-Letter Queue (DLQ) Volume**
    *   **Target:** <0.1% of total ingested rows end up in the DLQ.
    *   **Measurement:** Ratio of records in the MongoDB `sync_errors` collection flagged as DLQ (due to 3 consecutive `5xx` errors) versus total processed `SyncTasks`.
    *   **Why:** Tracks severe systemic or network instability that bypasses agentic resolution.

## Assumptions

### 1. Third-Party APIs & SaaS Infrastructure

*   **Intuit QuickBooks Online (QBO) API Stability & Quotas:**
    *   **Assumption:** Intuit will maintain its current API schema and continue to enforce its known concurrency limits (e.g., 500 requests per minute) rather than implementing more restrictive, undocumented throttling algorithms.
    *   **Impact if False:** If Intuit significantly restricts API access or fundamentally alters the definition of a `400 ValidationFault`, the Leaky Bucket queue will fail to prevent `429` rate-limit errors, and the crewAI micro-agents will fail to correctly intercept domain logic errors, breaking the core ingestion pipeline.
*   **Google Vertex AI SLA & Data Privacy:**
    *   **Assumption:** Google Vertex AI (`gemini-1.5-pro`) will maintain high availability and the Google Cloud Enterprise Data Processing Agreement (DPA) ensures that client financial data and PII are strictly explicitly excluded from being used for underlying LLM model training.
    *   **Impact if False:** If the service is unreachable, the system must degrade to manual mapping. More critically, if the DPA does not protect client data, enterprise CAS firms will be legally barred from using the product, completely invalidating the "Zero-Touch" business case.

### 2. User Behavior & Market Acceptance

*   **The Tolerance for "Agentic Overrides":**
    *   **Assumption:** Accounting firms (the CAS Manager / "Sarah" persona) will trust an AI agent to autonomously mutate financial payloads, provided there is a persistent, immutable audit log and a reliable "Physical Undo Switch."
    *   **Impact if False:** If the accounting industry's inherent risk aversion causes users to universally reject the `AgentResolving` automation step in favor of manual `RequiresIntervention` reviews, the system reverts to a manual workflow, defeating the core value proposition of decoupling headcount from revenue growth.
*   **Adoption of the "Ledger Dry Run":**
    *   **Assumption:** The P&L "Ledger Dry Run" simulation will provide sufficient psychological safety to cure "Ledger Anxiety" before the user clicks "Approve."
    *   **Impact if False:** If users do not trust the local dry run calculation, they will continue to manually verify large CSV files line-by-line prior to upload, destroying the <10 minute onboarding time-to-value target.

### 3. Data & Accounting Context

*   **Idempotency of Reversal (The Rollback Constraint):**
    *   **Assumption:** When executing the "One-Click Rollback", issuing exact inverse actions (e.g., `DELETE` or `VOID`) to the ERP API is sufficient to return the ledger to a mathematically pristine state (in terms of Balance Sheet and P&L impact), *assuming the accounting period remains open*. We assume users understand that internal, immutable ERP audit IDs for the deleted transactions will technically persist.
    *   **Impact if False:** If the ERP creates hidden sub-ledgers or locked tax-codes during ingestion that cannot be removed via standard API calls, or if the user expects all internal trace IDs to be wiped, the rollback will be deemed a failure, destroying the firm's trust in the product.
*   **ERP-Managed Currency Conversion:**
    *   **Assumption:** For multi-currency e-commerce data, the destination ERP (QBO/NetSuite) acts as the definitive source of truth for exchange rates based on the transaction date provided by the engine.
    *   **Impact if False:** If the ERP requires the engine to provide realized gain/loss calculations or specific forex rates, the engine would require a highly complex, redundant forex management layer to prevent out-of-balance errors during ingestion.
*   **Predictability of Historical CSV Structures:**
    *   **Assumption:** While client source data (e.g., Shopify, Amex) is messy, the variation in column headers is semantically recognizable by a generalized LLM (e.g., Vertex AI can reliably infer that "Client Bill-To" maps to `CustomerRef`).
    *   **Impact if False:** If legacy software exports are entirely encoded (e.g., columns named `F_11A` with proprietary integer codes), the Semantic Ingestion engine will fail, requiring bespoke, programmatic human intervention per client.

### 4. Scalability & Enterprise Future-Proofing

*   **Schema Agnosticism (Phase 3 Readiness):**
    *   **Assumption:** The crewAI resolution logic and MongoDB shadow-state schemas can be abstracted cleanly enough during Phase 1 (QBO) that connecting the NetSuite REST API in Phase 3 will not require a fundamental rewrite of the Python backend.
    *   **Impact if False:** If the error-resolution agents become deeply hardcoded to QBO-specific API quirks, the timeline and budget for enterprise expansion will exponentially inflate.
