# Executive Summary

## Problem Statement
In the highly competitive global creator economy, producing high-volume video ad variations is essential for discovering winning creatives. While AI avatar and voice generation (utilizing tools like HeyGen and ElevenLabs) successfully solve the human production bottleneck, they introduce a severe financial risk: blind, unoptimized generation at scale. Autonomously rendering hundreds of concurrent video variations where the vast majority fail to convert results in catastrophic API cost waste, completely eroding campaign margins. Furthermore, standard generative systems lack the closed-loop feedback required to systematically learn what drives conversions, leaving AdTech operators unable to incrementally improve Return on Ad Spend (ROAS) or rapidly scale winning creatives to new international markets.

## Target Audience & Key Stakeholders
* **Primary Users:** Elite affiliate marketers, performance agency directors, and growth-focused creative teams managing multi-million dollar global ad spends.
* **Key Stakeholders:** VP of Performance Creative, Media Buyers, Marketing Data Analysts, and Finance/RevOps leaders focused on unit economics and margin protection.

## Proposed Solution & Key Differentiators
The AI Avatar Affiliate Video Generator is designed to transcend basic mass-generation, functioning instead as a margin-optimized, self-learning performance engine. Orchestrated via CrewAI and seamlessly integrated into a Slack-native Minimum Viable Product (MVP), the platform serves as an automated performance creative director. 

Key product differentiators include:
* **Margin-Protecting "Pre-Flight" Workflow:** To eliminate API cost hemorrhaging, the system delivers lightweight storyboard payloads (localized Gemini text scripts, yt-dlp visual hook cues, and pacing data) directly into Slack threads. Computationally heavy and expensive rendering pipelines (RabbitMQ to ElevenLabs and HeyGen) are strictly gated and only triggered upon explicit manual approval via Slack Block Kit buttons.
* **Closed-Loop Performance Engine:** A specialized CrewAI "Performance Analyst" agent ingests post-campaign conversion data (CTR, CPA, ROAS) from Meta and TikTok via FastAPI webhooks. It analyzes which specific hooks, structures, and transition timings drove performance, logging these granular insights into an Obsidian knowledge base. Subsequent Gemini scriptwriter agents query this database, ensuring every new creative is mathematically informed by historically proven, channel-specific data.
* **Frictionless Global Arbitrage:** To capitalize on geographic market arbitrage, users can trigger a 1-click "Localize" Slack command on any winning creative. The asynchronous pipeline instantly translates the script with regional slang adaptations, synthesizes native-accented voiceovers, and adjusts avatar lip-sync for rapid deployment across global markets.

## Business Impact & Success Criteria
This platform is architected to rigorously protect margins while systematically accelerating campaign performance and global reach. Success will be measured against the following strict, measurable criteria:

* **Financial & Performance Metrics:** 
  * Achieve a **30% reduction** in wasted API rendering costs via the Slack Pre-Flight approval gate.

Fast-follower Phase 2 will introduce the FastAPI/Obsidian closed-loop learning architecture to begin compounding performance insights. ### 1.2 Technology Stack Decisions
*   **API Gateway & Webhooks:** `FastAPI` (Python). Chosen for native async support, Pydantic validation (critical for LLM structured outputs), and high throughput for inbound Meta/TikTok webhooks.
*   **Database:** `MongoDB Atlas`. Chosen for flexible schema evolution (API payloads from ad networks change frequently) and native TTL indexes for expiring unapproved storyboards.
*   **Message Broker:** `RabbitMQ`. Chosen over Kafka for simpler routing, exact DLQ (Dead Letter Queue) semantics, and delayed message exchanges (crucial for API backoff).
*   **AI Orchestration:** `CrewAI` framework. Used strictly for orchestrating sequential LLM tasks (Analyst -> Writer -> Translator) and managing Obsidian tool access.
*   **Knowledge Base:** `Obsidian` via EFS. Worker nodes mount an AWS Elastic File System (EFS) containing the Obsidian markdown vault. This allows standard markdown files to be read/written by agents while remaining compatible with the Obsidian desktop app for human auditing.
*   **Vision/Media Processing:** `yt-dlp` for reference downloads, `Clip + ViT` for local, fast image validation before sending to HeyGen (cost savings).

### 1.3 Data Flow: The Happy & Edge Paths

**Happy Path (Ideation to Render):**
1. User requests idea via Slack -> FastAPI receives command -> Queues `IdeationJob`.
2. CrewAI Node picks up job -> Queries Obsidian -> Calls Gemini -> Saves `VideoStoryboard` to Mongo (`Pending_Approval`).
3. FastAPI posts Block Kit message to Slack.
4. User clicks "Approve" -> Slack Webhook to FastAPI -> State changes to `Approved` -> Queues `RenderJob`.
5. Worker Node calls ElevenLabs -> Waits -> Calls HeyGen -> Waits -> Webhook callback updates Mongo to `Completed`.
6. FastAPI posts final MP4 to Slack.

**Nil/Empty Path (Insufficient Data):**
1. Analyst agent receives Meta webhook with $1.50 spend.
2. Rule engine flags `Spend < $50`.
3. State transitions immediately to `Discarded`. System explicitly does *not* invoke LLM (saves cost).

**Error Path (API Failure & Idempotency):**
1. User clicks "Approve" in Slack. Network blip causes Slack to retry.
2. FastAPI receives two identical requests. First request locks row and updates to `Approved`. Second request reads `Approved`, skips queuing, returns 200 OK (Idempotency).
3. HeyGen API returns 503. Worker node catches exception, NACKs message to RabbitMQ with a 60s delay header. Retries 3 times. On 4th fail, routes to DLQ, updates Mongo to `Failed`, alerts Slack channel.

## 5. Error Handling & Failure Modes

**Design Philosophy:** Zero silent failures. Every API drop must alert the Slack channel with a human-readable recovery path.

*   **API Timeouts / 5xx Errors (Minor/Major):** 
    *   *Handling:* RabbitMQ workers catch `httpx.HTTPError`. Message is NACK'd with `x-delay` headers (e.g., 10s, 30s, 60s). 
    *   *Limit:* Max 3 retries.
*   **HeyGen Profanity/Policy Rejection (Major):**
    *   *Handling:* Caught via HeyGen 400 Bad Request. Worker routes payload back to a specialized CrewAI "Sanitizer" agent to rewrite the flagged word phonetically. Attempts exactly 1 time before failing hard.
*   **Idempotency Failures (Critical):**
    *   *Handling:* Slack webhooks can fire multiple times if the backend takes > 3s to respond. FastAPI gateway *must* instantly acknowledge Slack via background tasks (`BackgroundTasks` in FastAPI), processing state transitions asynchronously. MongoDB atomic `findOneAndUpdate` ensures only one process transitions `Pending_Approval -> Approved`.
*   **Zombie Render Jobs (Major):**
    *   *Handling:* A job stuck in `Video_Rendering` because HeyGen dropped the webhook. A cron job runs every 10 minutes, sweeping jobs where `UpdatedAt` > 15 mins. Pings HeyGen `GET /v1/video_status` directly.

## 7. Security & Trust Boundaries

*   **Authentication & Webhooks:**
    *   Slack: Validate `X-Slack-Signature` using HMAC-SHA256 and `SLACK_SIGNING_SECRET`. Reject any request older than 5 minutes to prevent replay attacks.
    *   Meta/TikTok: Validate `X-Hub-Signature`.
    *   HeyGen Callback: Validate using a cryptographically secure random token generated per `RenderJob` and passed in the webhook URL (e.g., `/api/v1/webhooks/heygen/{secure_token}`).
*   **Data Classification & PII:**
    *   User voice samples and images are considered **Sensitive/PII**.
    *   Storage: S3 buckets must be configured with `Block Public Access` and objects served via presigned URLs with 15-minute expirations for browser/worker access.
*   **Attack Surface Mitigation:**
    *   Prompt Injection: CrewAI system prompts must strictly delineate user input (`RawPrompt`) from system instructions. Output schemas must strictly enforce enum types and string bounds.

## 8. Deployment & Rollout

### 8.1 Deployment Sequence
We utilize a containerized deployment strategy (Docker, ECS/EKS).
1. **Database:** Apply MongoDB index updates via Motor script.
2. **Workers:** Deploy updated RabbitMQ consumer nodes (CrewAI, Render, Analyst). *Note: Workers must deploy before the API to ensure new job types can be processed.*
3. **API Gateway:** Deploy the FastAPI instances (Rolling Update).
4. **Web UI:** Deploy React static build to CDN (CloudFront/Vercel).

### 8.2 Rollback Plan
1. If API error rate spikes > 1% post-deploy: Revert FastAPI ECS service to previous image tag.
2. If Workers fail to process jobs: Pause RabbitMQ queues -> Revert Worker ECS service -> Unpause queues.
3. Database migrations (indexes) are generally forward-compatible. No immediate rollback required unless a unique constraint breaks legacy data (prevented via staging E2E tests).

## Problem Statement

The historical bottleneck in global performance marketing and affiliate AdTech has always been creative production. Producing enough high-quality video variations to find a "winning" ad required extensive human capital, agency retainers, and weeks of lead time. Recent advancements in generative AI—specifically high-fidelity avatar and voice synthesis APIs—have entirely removed this bottleneck. However, solving the production problem has inadvertently introduced a far more dangerous threat to enterprise AdTech: the financial catastrophe of blind, unoptimized generation at scale.

We are no longer facing a content deficiency; we are facing a predictability and margin-erosion crisis. The core problems driving this crisis can be broken down into four specific pillars:

**1. The "Creative Casino" and API Cost Hemorrhage**
Current AI generation workflows operate on a "spray and pray" methodology. Media buyers generate hundreds of concurrent video variations, hoping a fraction will resonate with the market. However, premium generation APIs (such as HeyGen and ElevenLabs) are computationally heavy and highly expensive. In a standard high-volume campaign, up to 95% of generated creatives will fail to meet target Return on Ad Spend (ROAS). Because there is no cost-gating mechanism prior to rendering, agencies are paying full price to produce failures. The wasted API compute costs completely cannibalize the profit margins generated by the 5% of winning ads. We are treating expensive infrastructure like a slot machine.

**2. Visual Data Blindness and the Broken Feedback Loop**
Current generative AI tools are entirely decoupled from market reality. They exist purely as execution layers, possessing no memory of historical campaign performance. Crucially, they suffer from "Visual Data Blindness"—when an ad fails on Meta or TikTok, the system cannot programmatically deconstruct *why* it failed. It cannot analyze the video frames to determine if the 3-second visual hook was weak, if the pacing dragged, or if the script structure was suboptimal. Consequently, creative strategists are forced to manually guess the variables of success. Without an automated system to deconstruct visual/audio payloads, ingest post-campaign conversion data, and mathematically inform the next generation of scripts, the production cycle remains stagnant, preventing any compounding lift in creative performance.

**3. Identity Fragmentation Across Media Channels**
Scaling a campaign across multiple media channels (e.g., TikTok, YouTube Shorts, Threads, X) requires distinct personas tailored to each platform's unique culture and format. Currently, maintaining persona consistency is a manual, high-friction process. Taking a single creative concept and manually adapting the avatar’s visual styling, voice characteristics, and script tone across multiple platforms introduces massive operational overhead. This identity fragmentation prevents rapid multi-channel deployment, leaving revenue on the table as competitors move faster to dominate emerging platforms.

**4. Dashboard Fatigue and Synchronous Idle Time**
The day-to-day workflow of an elite media buyer has devolved into an exercise in "dashboard fatigue." The iterative process of refining a raw idea into a high-probability script requires exhaustive context-switching across disjointed toolchains—bouncing between LLM chat interfaces for scripting, audio platforms for voice testing, avatar platforms for visual rendering, and ad managers for deployment. Furthermore, this workflow is crippled by a "Synchronous Bottleneck." Humans are forced to wait idly while heavy video generation APIs render content before they can review, approve, or localize it. This back-and-forth prompt engineering and synchronous waiting takes days of manual coordination, leading to high cognitive load, prompt fatigue, and drastically reduced operational throughput.

**Impact Quantification**
The combination of zero cost-control, disconnected feedback loops, and severe operational friction is actively destroying unit economics. For an agency deploying $1M+ in monthly ad spend, unoptimized API generation easily results in tens of thousands of dollars in wasted operational expenditure (OpEx) monthly. Concurrently, the operational latency caused by disjointed toolchains and synchronous iteration bottlenecks prevents agencies from rapidly localizing and multiplying their revenue on proven assets before ad fatigue sets in.

**Why Now?**
Ad platforms are rapidly becoming saturated with commoditized, high-volume AI content, which is steadily driving up media buying costs (CPMs). Brute-forcing creative testing by simply generating *more* videos is no longer a viable financial strategy. To survive and scale in this compressing market, elite AdTech operators must immediately transition from unconstrained, high-volume generation to high-probability, margin-protected generation unified within a frictionless, asynchronous operational layer.

## Edge Cases

### 1. Concurrent Interaction & Setup Anomalies

**EC-101: Simultaneous Slack Approvals**
* **Scenario:** Multiple users in the same Slack channel (e.g., the Media Buyer and the VP of Performance) simultaneously click the `[Approve & Render]` Block Kit button on the same storyboard payload.
* **System Behavior:** The system MUST utilize an atomic `findOneAndUpdate` operation in MongoDB on the `VideoStoryboard` document. The first request successfully updates the state from `Pending_Approval` to `Approved` and triggers the `RenderJob`. Subsequent requests evaluate the state as already `Approved`, skip queueing duplicate RabbitMQ jobs, and return an ephemeral Slack message: *"This storyboard has already been approved."*

**EC-102: Rapid 1-Click Localization Triggers**
* **Scenario:** A user rapidly clicks the "Localize to LATAM" button multiple times before the Slack UI updates.
* **System Behavior:** The system MUST generate a deterministic hash based on the `StoryboardId` and the `TargetMarket`. The FastAPI endpoint MUST check for an active `LocalizationRequest` with this hash. If one exists, the system drops the duplicate request and returns an HTTP 200 OK without queueing a new job.

**EC-103: Identity Modeling Rejection**
* **Scenario:** A user's uploaded avatar assets pass local Clip+ViT validation but are subsequently rejected by the HeyGen or ElevenLabs asynchronous training process (e.g., due to background audio noise or unsupported face angles).
* **System Behavior:** The system intercepts the webhook error from the provider. It immediately resets the `MediaChannel` modeling state to `Failed` and sends an alert in the Slack channel where the original upload occurred, detailing the specific provider rejection reason and providing a Block Kit button to restart the upload process.

**EC-104: Cross-Channel Identity Validation**
* **Scenario:** A user attempts to render a video using an Avatar Profile that is not explicitly mapped to the active Media Channel.
* **System Behavior:** Before the `RenderJob` is queued, the system MUST verify the `AvatarID`-to-`ChannelID` mapping in MongoDB. If the mapping is invalid, the system blocks the render and alerts the user of the configuration mismatch in Slack to prevent wasted API spend on the wrong brand persona.

### 2. Generative AI & Tool Boundary Failures

**EC-201: LLM Generative Length Inflation**
* **Scenario:** The Gemini Scriptwriter agent hallucinates or ignores the strict 15-60 second duration constraint, returning a script that will require 2 minutes of audio generation.
* **System Behavior:** Before presenting the Pre-Flight storyboard, the CrewAI orchestration layer MUST programmatically estimate the audio duration based on word count. If the estimate exceeds 65 seconds, it triggers an internal retry loop (max 2 attempts) with a stricter prompt constraint. If it persistently fails, the system returns an error to Slack: *"Generated script exceeded duration constraints. Please refine your idea prompt."*

**EC-202: Slang Translation Hallucination & Policy Violation**
* **Scenario:** During the geographic localization flow, Gemini translates an English hook into a regional slang phrase that unintentionally violates HeyGen or ElevenLabs' content moderation policies (e.g., generating a banned word).
* **System Behavior:** The system MUST intercept the HTTP 400 Bad Request (Policy Violation) from the 3rd-party API. The worker node routes the payload back to a specialized CrewAI "Sanitizer" agent with the specific error context to rewrite the flagged phrase using synonyms or phonetic alternatives. This happens transparently to the user, attempting one retry before explicitly failing the job and alerting Slack.

**EC-203: Invalid Reference URL for Hook Deconstruction**
* **Scenario:** A user provides a YouTube/TikTok URL for hook extraction, but the video is private, geo-blocked, or deleted, causing the `yt-dlp` tool to fail.
* **System Behavior:** The `IdeationJob` catches the `yt-dlp` exception. Instead of failing the entire ideation process, the system degrades gracefully. It proceeds with script generation using historical Obsidian best practices and adds a disclaimer to the Slack storyboard: *"Note: Reference URL unavailable; script generated using historical best practices."*

### 3. Data Integrity & Feedback Loop Anomalies

**EC-301: Delayed Attribution / Ad Network Webhook Lag**
* **Scenario:** Meta or TikTok delays sending conversion webhooks by 48+ hours, or sends them in massive batches, long after the video was published.
* **System Behavior:** The FastAPI webhook ingestion endpoint MUST accept historical conversion data regardless of the timestamp. To prevent real-time data thrashing, the `Performance Analyst` agent runs as a scheduled cron job to periodically sweep the MongoDB persistence layer, aggregating delayed data and updating the Obsidian knowledge base in bulk.

**EC-302: Organic Virality Skewing ROAS Data**
* **Scenario:** A generated video goes organically viral on TikTok, resulting in massive conversions with zero recorded ad spend. If ingested blindly, the ROAS calculation approaches infinity, permanently skewing the Obsidian learning weights.
* **System Behavior:** The Performance Analyst agent MUST evaluate the `Spend` metric. If `Conversions > 0` but `Spend` falls below the $50 threshold, the agent categorizes the data as an "Organic Trend." This data is logged for qualitative insights but is explicitly excluded from the mathematical weights used for future script generation to protect "Quant" predictability.

**EC-303: Knowledge Base Parsing Failure**
* **Scenario:** The Analyst agent writes malformed Markdown or an unparseable structure to Obsidian during the feedback loop.
* **System Behavior:** If the Scriptwriter agent encounters an unparseable or corrupted file in a channel's Obsidian vault, it MUST ignore the specific corrupted entries, log a `Parsing_Error` for engineering review, and gracefully fall back to the global high-performance template.

**EC-304: Knowledge Base Cold Start**
* **Scenario:** A user generates content for a brand-new Media Channel with zero historical data in its specific Obsidian vault.
* **System Behavior:** The Scriptwriter agent detects the empty vault and automatically pulls "Seed Strategy" variables from a system-level Obsidian volume containing industry-standard winning hooks for that specific platform to ensure the first generation is not a blind guess.

### 4. Network & Infrastructure Edge Cases

**EC-401: Slack API Rate Limiting on Delivery**
* **Scenario:** The RabbitMQ render worker completes the HeyGen video, but attempting to upload the MP4 to Slack results in an HTTP 429 (Too Many Requests) from the Slack API.

If the Slack upload fails after 3 attempts, the worker uploads the final MP4 to the system's S3 bucket, generates a pre-signed URL (7-day expiration), and posts a plain-text message to the Slack thread containing the download link.

**EC-402: C2PA Metadata Injection Failure**
* **Scenario:** The HeyGen API returns a valid MP4, but the internal C2PA CLI tool fails to attach the cryptographic metadata tag.
* **System Behavior:** The system MUST NOT deliver untagged media to the user. The worker node flags the job as a `C2PA_Failure`, routes the message to the DLQ, and alerts the Slack channel: *"Video rendered successfully, but failed compliance tagging. Engineering has been notified."*

* **Scenario:** A render worker receives an "Insufficient Credits" or "Payment Required" HTTP 402 error from HeyGen or ElevenLabs.
* **System Behavior:** The system MUST immediately pause the specific provider's RabbitMQ queue to prevent cascading failures. **EC-501: Multi-Platform Aspect Ratio Conflict**
* **Scenario:** A user attempts to localize or duplicate a highly specific 9:16 (vertical) script/render to a media channel whose base profile requires a 16:9 (horizontal) format.
* **System Behavior:** The system MUST cross-reference the active `MediaChannel` platform type with the render's intended dimensions before queuing the job. If a conflict is detected, the system MUST alert the user via Slack to the format conflict and require explicit confirmation to proceed before incurring API costs.

### 1. Error Taxonomy & Global Handling Strategy

The system enforces a "Zero Silent Failures" philosophy. All errors are categorized to protect API margins and ensure the "Quant Fund" maintains a perfectly accurate ledger of sunk costs. Every error handled below explicitly triggers a `System_Event_Broadcast` to the FastAPI/MongoDB event log, ensuring real-time visibility across both the Slack Command Center and the React Web UI Dashboard.

| Error Category | System / HTTP Code | Description | Global Handling Strategy |
| :--- | :--- | :--- | :--- |
| **API & Integration** | 5xx | Provider timeouts or internal server errors (HeyGen/ElevenLabs). | Async RabbitMQ retry (x-delay: 10s, 30s, 60s). Max 3. |
| **Authentication** | 401 | Provider API key revoked, expired, or balance-restricted. | Immediate Queue Pause. Route to VP for credential update. |
| **Cost & Billing** | 402 | External provider wallet empty or channel budget threshold breached. | Immediate Queue Pause. Route to VP for manual wallet top-up. |
| **Content & Policy** | 400 | Moderation rejection of generated script by 3rd-party providers. | Route to CrewAI "Sanitizer" agent (1 attempt, 45s TTL). |
| **Data & Validation** | 422, 404 | Missing webhook data, unparseable Obsidian markdown, or invalid asset uploads. | Graceful degradation (Seed Vault fallback) or Hard Fail. |
| **Infrastructure** | 429 | Slack API rate limiting during final payload delivery. | Exponential backoff. Fallback to S3 Pre-signed URL. |

### 2. Third-Party API & Integration Errors

**ERR-201: Provider Timeout or Service Unavailable**
* **Trigger:** Worker node receives a 5xx error from HeyGen or ElevenLabs.
* **System Handling:** The worker node NACKs the message to RabbitMQ with an exponential backoff header. If the 3rd retry fails, the job moves to the Dead Letter Queue (DLQ). **Crucially:** Any sunk costs from preceding successful API calls in the chain (e.g., ElevenLabs audio success before a HeyGen failure) MUST be explicitly logged to MongoDB and the Obsidian `Waste_Report` to maintain accurate ROAS calculations.
* **Resumption Logic:** If a user triggers a retry from a partially successful DLQ state, the worker MUST check for existing intermediate artifacts in AWS S3 (e.g., the previously generated audio file) and resume the workflow from the exact point of failure to prevent double-billing of API credits.
* **User/Platform UX:** 
  * **Slack:** *"Provider Error: HeyGen is unreachable. Job paused. [Retry Job] | [Cancel & Refund Cost]."*
  * **Dashboard:** Flags the job row as `Provider_Outage` in the Active Renders view.

**ERR-202: Slack Rate Limiting (HTTP 429)**
* **Trigger:** The system attempts to upload a rendered MP4 to a Slack channel but hits the Slack API rate limit.
* **System Handling:** The system implements a local exponential backoff for the Slack payload (max 3 attempts). On the 3rd failure, the system uploads the final MP4 to the system's AWS S3 bucket and generates a pre-signed URL with a 15-minute expiration.
* **User/Platform UX:** 
  * **Slack:** The user receives a plain-text message: *"Your video is ready, but Slack is rate-limiting large file uploads. Download your video securely here: [S3 Link]."*

### 3. Financial & Authentication Errors

**ERR-301: External Provider Credit Exhaustion (HTTP 402)**
* **Trigger:** ElevenLabs or HeyGen API returns an "Insufficient Credits" response when a `RenderJob` initiates.
* **System Handling:** The system immediately halts the specific provider's RabbitMQ consumer queue to prevent cascading job failures and flags the specific `RenderJob` as `Billing_Hold`.
* **User/Platform UX:** 
  * **To Media Buyer (Slack):** *"Render Paused: The external API wallet is empty. The VP of Performance has been notified."*
  * **To VP (Slack & Dashboard):** High-priority Alert: *"Critical: HeyGen wallet exhausted. [Number] render jobs are currently blocked. [Click here to Top Up HeyGen Wallet]."*

**ERR-302: Provider Authentication Failure (HTTP 401)**
* **Trigger:** ElevenLabs or HeyGen API returns an "Unauthorized" response due to an expired or revoked API key.
* **System Handling:** The system immediately pauses the affected RabbitMQ provider queue and logs a `Credential_Failure` event.
* **User/Platform UX:** 
  * **To VP (Slack & Dashboard):** High-priority Alert: *"Critical: HeyGen API key rejected (401). [Number] jobs are paused. Please update the API credentials in the system settings."*

### 4. Generative AI & Content Errors

**ERR-401: Content Policy Rejection**
* **Trigger:** HeyGen or ElevenLabs rejects a generated script payload due to policy violations (e.g., restricted slang generated during localization).
* **System Handling:** The worker node intercepts the HTTP 400 Bad Request. It routes the payload to the internal CrewAI "Sanitizer" agent to rewrite the flagged phrase phonetically or using a synonym. 
* **Constraint Limits:** The Sanitizer agent is restricted to exactly **one (1) attempt** and a strict **45-second Time-to-Live (TTL)** to protect the overall 5-minute render SLA. If the retry is rejected again or the TTL expires, the job triggers a Hard Fail, and any partial sunk costs are logged to the Obsidian `Waste_Report`.
* **User/Platform UX:** Handled transparently if successful. If the retry/TTL fails, notify the user: *"Error: The script violates provider safety policies. Please adjust the phrasing manually."*

**ERR-402: Unparseable Obsidian Knowledge Base Data**
* **Trigger:** The Gemini Scriptwriter agent attempts to query the Obsidian EFS volume but encounters malformed Markdown or corrupted historical ROAS data.
* **System Handling:** The system catches the `Parsing_Error` exception. It MUST NOT fail the ideation process. The system gracefully degrades by ignoring the corrupted channel-specific file and pulling from the global "Best Practice" Seed Vault to maintain generation continuity.
* **User/Platform UX:** The system appends a warning to the Pre-Flight storyboard payload in Slack: *"(Warning: Channel-specific historical data was unparseable. This script was generated using global best-practice benchmarks. Engineering has been notified.)"*

### 5. Input & Validation Errors

**ERR-501: Invalid Identity Modeling Assets**
* **Trigger:** A user uploads avatar training images via Slack that fail the local Clip+ViT validation (e.g., multiple faces detected, resolution < 720p).
* **System Handling:** The system triggers a Hard Fail, rejecting the payload before initiating any external API calls, and updates the `AvatarProfile` state to `Failed`.
* **User/Platform UX:** The system immediately replies in the Slack thread: *"Asset Validation Failed: Resolution < 720p. Please upload high-quality reference images featuring only the target avatar."*

**ERR-502: Missing or Delayed Webhook Data (Zombie Jobs)**
* **Trigger:** A `RenderJob` is queued and successfully accepted by HeyGen, but the system does not receive the completion webhook within 15 minutes.
* **System Handling:** A background cron job runs every 10 minutes sweeping the MongoDB `RenderJobs` collection for jobs in the `Video_Rendering` state where `UpdatedAt > 15 mins`. The system performs a direct `GET /v1/video_status` poll to the HeyGen API to reconcile the state. If the direct poll confirms the job failed mid-render, the system explicitly logs the incurred cost to the `Waste_Report` and triggers the standard **ERR-201** UX recovery loop.
