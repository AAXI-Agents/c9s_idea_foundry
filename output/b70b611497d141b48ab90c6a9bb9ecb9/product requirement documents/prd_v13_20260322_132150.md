# Product Requirements Document

## Executive Summary

**1. Problem Statement**
In the highly competitive Ho Chi Minh City (HCMC) real estate market, the 6-12 story building segment presents a lucrative opportunity that is severely bottlenecked by two distinct frictions: an opaque analytical feasibility process and a fragmented communication pipeline. Currently, developers spend weeks and thousands of dollars navigating localized data fragmentation—such as 1/500 zoning approvals, strict Floor Area Ratios (FAR), and "Pink Book" conversion fees—only to face a secondary bottleneck in packaging these complex financial metrics for diverse audiences. Whether pitching high-density ROI to institutional investors or broadcasting project insights to social media followers (TikTok, Facebook) to raise capital, traditional analysis is too slow. By the time feasibility studies are completed and formatted for these specific channels, premium parcels are frequently lost to faster buyers. 

**2. Target Audience & Key Stakeholders**
*   **Primary Users:** On-the-go boutique real estate developers, family office investment directors, local syndicators, and real estate "Content Creators" operating in Vietnam. These mobile-first users require immediate intelligence and the ability to seamlessly broadcast tailored insights across multiple media channels without sitting at a desk.
*   **Key Stakeholders:** PropTech engineering teams, third-party urban planning data providers, institutional investment partners, and social media audience networks.

**3. Proposed Solution & Key Differentiators**
The HCMC "Virtual Analyst" is an event-driven, multi-agent AI engine that evaluates real estate parcels and delivers synthesized video briefings via a Slack-first integration. It bridges the gap between deep PropTech analytics and media publishing by acting as an "Insight-to-Broadcast" engine.

*   **Identity & Multi-Channel Management:** Users initialize their profile via a required Gmail/Email authentication linked to their Slack ID to ensure data persistence across future Web/Mobile platforms. Using a `/manage-channels` command, users define targeted media destinations (e.g., "TikTok," "X," "Investor Group"). The `/setup-avatar --channel` workflow allows users to upload 5-10 photos and voice clips to generate distinct, channel-specific digital twins (e.g., a "Professional Suit" avatar for investors vs. a "Casual" avatar for TikTok) powered by HeyGen and ElevenLabs.
*   **Dual-Mode Insight Generation:** Dropping an HCMC address or pin into Slack triggers a FastAPI webhook and RabbitMQ queue. A specialized CrewAI backend (Zoning, Financial, and Market Comp agents) analyzes the parcel while utilizing `yt-dlp` to pull viral real estate trends for context. The Orchestrator Agent then generates a Dual-Mode Script: *Mode A (Internal)* featuring high-density ROI and regulatory metrics, and *Mode B (External)* featuring a 15-60 second hook-driven script optimized for social publishing.
*   **Iterative Asynchronous Delivery:** To manage expensive API costs and Slack’s 3-second webhook timeout, the workflow is decoupled. *Stage 1 (Draft Mode)* instantly returns a text-based feasibility summary and proposed scripts. If 1/500 data is missing, a "Data-Entry Fallback" prompts the user to input FAR manually. *Stage 2 (Video Render)* is initiated only upon user approval. Leveraging CrewAI Event Listeners, the final rendered video is delivered back to the Slack thread as a secure, cloud-hosted Signed URL, entirely bypassing Slack's large-file payload limits.

**4. Business Impact & Success Criteria**
The core business objective is to provide a massive competitive advantage by transforming weeks of opaque due diligence into an instant, multi-channel broadcast asset. Success for the MVP will be measured by the following SMART criteria:
*   **System Performance (Latency):** Deliver the initial text-based Dual-Mode feasibility draft within 45 seconds (p95), and asynchronously deliver the final video Signed URL within 8 minutes (p95) post-user approval.
*   **Cost Management:** Enforce a strict "Daily Credit" limit per user via the Slack bot to throttle high-cost HeyGen/ElevenLabs API usage, strictly adhering to the "low cost to free" operational guardrails.
*   **Value Validation (Analytics):** Achieve a minimum 40% "Conversion from Script to Video" ratio, proving the AI's financial modeling and scripting deliver enough value to justify the user executing the final render.
*   **Risk Mitigation (Trust):** Attain 100% adherence to LLM confidence scoring protocols. If the Zoning Agent's confidence falls below 90%, the draft must explicitly state the regulatory risk. Concurrently track an "Agent Confidence Frequency" metric to identify specific HCMC districts requiring manual data augmentation to prevent silent hallucinations. All workflow states, identity maps, and financial calculations must be reliably persisted in MongoDB Atlas.

### 1.2 Technology Stack Rationale

| Component | Technology | Rationale |
| :--- | :--- | :--- |
| **API Gateway** | FastAPI (Python) | High-performance async I/O. Native Pydantic validation guarantees structured API contracts. Ideal for ML/LLM orchestration. |
| **Database** | MongoDB Atlas | Document-store flexibility is required for highly variable AI outputs (`raw_agent_logs`, varied `financial_data` payloads). |
| **Message Broker** | RabbitMQ | Pub/sub model guarantees at-least-once delivery for AI jobs. Essential for decoupling Slack webhooks from slow multi-agent generation tasks. |
| **AI Framework** | CrewAI | Native state sharing and inter-agent routing. Specifically using CrewAI Event Listeners to bind directly to RabbitMQ queues. |
| **LLM Engine** | Gemini Flash/Pro | Fast, high-context-window models for reasoning and data extraction. |
| **Knowledge Base** | Obsidian (Markdown) | Lightweight, easily version-controlled local RAG target for 1/500 zoning regulations. |
| **Media AI** | HeyGen & ElevenLabs | Best-in-class avatar rendering and voice cloning. Expensive and slow, requiring strict rate-limiting and async polling via webhooks. |
| **Vision Model** | Clip + ViT (Local) | Zero-shot visual classification to gatekeep bad user photos *before* sending them to paid external APIs. |

## 2. Component Breakdown

### 2.1 Slack Ingestion & Routing Service
*   **Purpose**: Terminate TLS, validate webhook signatures natively, and extract addresses before pub/sub.
*   **Dependencies**: Slack API, Gemini Flash (for lightweight extraction), RabbitMQ.
*   **API Contract**:
    *   `POST /api/v1/webhooks/slack/events`
    *   Headers: `X-Slack-Signature`, `X-Slack-Request-Timestamp`
    *   Payload: Slack Events API JSON.
    *   Response: `200 OK` (Empty body).

### 2.2 Avatar Provisioning Service
*   **Purpose**: Manages multi-tenant media channels and AI avatar pipelines.
*   **Dependencies**: Local Clip+ViT, HeyGen API, ElevenLabs API, S3/CloudFront.
*   **State Machine (`AvatarProfile`)**:

```text
       [Draft]
          | (User submits images & voice)
          v
    [Validating] -------------------------------+
          | (Clip+ViT passes)                   | (Clip+ViT fails: blurry/occluded)
          v                                     |
     [Training] <-----------------------+       |
          | (API calls dispatched)      |       v
          |                             |   [Failed]
          | (Webhook Success)           |       | (User re-submits)
          v                             +-------+
      [Active]
```

### 2.3 CrewAI Feasibility Engine
*   **Purpose**: The core underwriting brain. Resolves coordinates to FAR, zoning risk, and Cap Rate.
*   **Dependencies**: RabbitMQ, Gemini Pro, Obsidian Local RAG.
*   **State Machine (`FeasibilityRequest` & `VideoGenerationJob`)**:

```text
   [Received] --> (Parse) --> [Queued] --> (RabbitMQ) --> [Analyzing]
                                                              | (CrewAI Swarm)
   +----------------------------------------------------------+
   |
   v
[Scripting] --> (Orchestrator) --> [Rendering] (Wait for HeyGen Webhook)
                                       |
               +-----------------------+-----------------------+
               | (Webhook OK)                                  | (Timeout > 5m)
               v                                               v
          [Delivered]                                [Delivered (Fallback)]
       (Video in Slack)                             (Text-only in Slack)
```

## 5. Error Handling & Failure Modes

| Component / Process | Failure Mode | Mitigation / Circuit Breaker Strategy | Severity |
| :--- | :--- | :--- | :--- |
| **Slack Webhook** | Request dropped due to timeout (>3s) | **Ack First, Process Later.** FastAPI immediately returns 200 OK after validating HMAC and drops payload to RabbitMQ. | Critical |
| **LLM RAG (Zoning)** | Hallucinates zoning law | **Confidence Guardrail.** Prompt explicitly requires confidence score. Output validator forces status to `REVIEW` and prepends 🔴 warning if $< 0.80$. | Critical |
| **HeyGen API** | API down or render takes $> 5$ mins | **Graceful Degradation.** Background task sweeps `VideoGenerationJob`. If `updated_at` $> 5$ mins in `Rendering`, force transition to `Delivered (Fallback)` and send Text+Emoji summary to Slack. | Major |
| **ElevenLabs API** | Voice generation fails | **Retry Policy.** 3 retries with exponential backoff (2s, 4s, 8s). If failure persists, trigger fallback text generation. | Major |
| **User Image Upload** | Poor lighting/occluded face | **Pre-flight Vision Check.** Clip+ViT zero-shot classification intercepts bad images. Rejects request with specific actionable feedback to user *before* API spend. | Minor |
| **LLM Over-scripting**| Script exceeds 60 seconds | **Content Truncation.** Output parser counts words. If $> 80$ words, force single prompt retry asking for strict truncation. | Minor |

## 6. Test Strategy

### 6.1 Test Pyramid
*   **Unit Tests (60%)**:
    *   Strict coverage on Slack HMAC validation (simulate bad timestamps/signatures).
    *   Coverage on LLM Pydantic parsers (ensuring string vs. float extraction).
    *   *Critical Path:* The "Zero Silent Failures" confidence scorer. Feed mock JSON with `0.79` confidence to ensure the state machine forcibly triggers the `REVIEW` override.
*   **Integration Tests (30%)**:
    *   RabbitMQ Pub/Sub: Assert that a published `request.feasibility.new` event updates the MongoDB `FeasibilityRequest` status to `Queued`.
    *   CrewAI Event Listener binding tests.
*   **E2E Tests (10%)**:
    *   Mocking an entire Slack webhook -> MongoDB -> mock CrewAI -> Webhook Callback -> mock Slack `postMessage`.

### 6.2 Edge Case Matrix
*   *Empty Map Pin:* User types "check this out" with no location. System must reply politely asking for an address/pin.
*   *Out of Bounds Location:* User drops a pin in Hanoi. System must reject and state "HCMC only."
*   *Duplicated Webhook:* Slack sends the same event twice. Database unique index on `slack_thread_ts` must catch and drop the duplicate gracefully.

## 7. Security & Trust Boundaries

*   **Authentication & Authorization**:
    *   Web Portal: JWT tokens with short expiration (15m) and HTTP-only refresh tokens.
    *   Slack Webhook: Mandatory HMAC-SHA256 signature validation against `SLACK_SIGNING_SECRET`. Reject payloads where `X-Slack-Request-Timestamp` is older than 5 minutes to prevent replay attacks.
    *   External Webhooks (HeyGen): Requires custom `Bearer` token verification before accepting job completion status.
*   **Data Classification**:
    *   User face photos and voice samples are considered **Sensitive PII**.
    *   Assets stored in S3 must be private. System uses temporary **Pre-Signed URLs** (expiring in 1 hour) when passing assets to HeyGen/ElevenLabs APIs or serving video back to Slack.
*   **Resource Protection**:
    *   API routes rate-limited via Redis (e.g., 60 req/min for web UI).
    *   Daily credit limit enforced at the database level to prevent API bankruptcy via malicious/spam Slack requests.

## 8. Deployment & Rollout

### 8.1 Deployment Sequence
We will use Dockerized containers orchestrated via AWS ECS/Fargate or a managed K8s cluster.

1.  **Phase 1**: Deploy MongoDB Atlas schema changes & Redis instance.
2.  **Phase 2**: Deploy RabbitMQ cluster.
3.  **Phase 3**: Deploy Core FastAPI web services (Blue/Green).
4.  **Phase 4**: Deploy CrewAI Worker instances (Canary). Scale strictly based on RabbitMQ queue depth.

### 8.2 Feature Flags
Managed via environment variables or LaunchDarkly:
*   `FF_ENABLE_HEYGEN_RENDER`: If false, forces the fallback Text-only Slack delivery path. Useful for saving costs during internal testing.
*   `FF_ENABLE_STREETVIEW_VIBE`: Toggles the Google Street View API lookup.
*   `FF_ENABLE_YTDLP_TRENDS`: Toggles the background scraping job.

### 8.3 Rollback Plan
1.  **API Failure**: Shift load balancer traffic back to Blue environment.
2.  **Worker Failure (Bad Prompt/Hallucinations)**: Scale down CrewAI workers to 0. Drain RabbitMQ `proptech.events` queue to a Dead Letter Queue (DLQ). Revert CrewAI image, redeploy, and replay DLQ.
3.  **Database Corruption**: Point-in-time recovery via MongoDB Atlas (RPO < 5 minutes).

### 9.3 Debugging Guide: Stalled Requests
If a user complains "I dropped a pin but got no reply":
1.  Search logs for `correlation_id == slack_thread_ts`.
2.  Check `FeasibilityRequest.status`.
    *   If `Queued`: RabbitMQ workers are dead or queue is backed up.
    *   If `Analyzing`: CrewAI is hanging. Check Gemini API quotas.
    *   If `Rendering`: HeyGen webhook failed. Check Cron fallback sweeper logs.

## Problem Statement

**Current State Description**
In the hyper-competitive Ho Chi Minh City (HCMC) real estate market, boutique developers, family offices, and local syndicators primarily target the lucrative 6-12 story building segment (serviced apartments, mini-hotels, and grade-B office spaces). Currently, the deal syndication lifecycle—from discovering a raw plot of land to successfully pitching an underwritten deal to investors—is a deeply fragmented, entirely manual, and desk-bound process. Developers must rely on disjointed municipal contacts, bloated spreadsheets, and manual video production to close a single deal.

**Pain Points**
*   **Opaque Regulatory Underwriting:** To determine basic plot viability, developers must manually navigate highly localized, fragmented data to verify 1/500 zoning approvals, calculate strict Floor Area Ratios (FAR), and estimate unpredictable "Pink Book" (Land Use Right Certificate) conversion fees. This process is highly prone to human error and hidden regulatory risks.
*   **The Syndication Lag:** Underwriting risk and syndicating the opportunity to capital partners are treated as two disconnected workflows. After spending weeks validating the numbers, developers waste additional days manually translating dense feasibility spreadsheets into digestible, professional pitches for Limited Partners (LPs) or retail investors across varying media channels.
*   **Workflow Context Mismatch:** Deal inception happens in the field. Real estate syndicators are constantly in transit between physical sites, yet existing analytical and capital-raising tools require them to be stationary at a desk. This creates a severe operational bottleneck right at the moment of discovery.

**Impact Quantification**
*   **Sunk Financial & Time Costs:** Developers routinely burn $2,000+ and spend 3 to 4 weeks executing a single manual feasibility study per address.
*   **Fatal Loss of Deal Velocity:** The current process creates a 30-day "discovery-to-pitch" timeline. Because of this extreme time-to-decision lag, boutique developers consistently lose premium parcels to tier-1 institutional funds that possess faster in-house analytical teams. 
*   **Diminished Capital Conversion:** When complex zoning constraints and cap-rate breakdowns are poorly communicated due to rushed, manual pitch creation, investor trust drops, directly reducing the rate of committed capital.

**Why Now?**
The HCMC real estate market is experiencing an aggressive land grab in the mid-tier commercial segment, accompanied by rapid yield compression. Institutional capital is moving faster than ever. For boutique syndicators, capital velocity and deal conviction are no longer just competitive advantages—they are prerequisites for survival. If local developers cannot immediately match the underwriting speed and capital-markets polish of 50-person institutional firms while standing on the actual dirt lot, they will be permanently priced out of the market.

## Functional Requirements

**1. Slack-Native User Provisioning & Identity**

**FR-000: Slack-Native Onboarding & Identity Linking**
*   **Priority:** SHALL
*   **Description:** The system shall provide a `/setup-avatar` Slack command to initialize a user profile, linking their Slack identity to a persistent email account without requiring a web portal.
*   **Acceptance Criteria:**
    *   **Given** an unregistered user executes `/setup-avatar` in Slack,
    *   **When** the system receives the command,
    *   **Then** it shall generate a secure OAuth "Magic Link" sent via Slack DM to bind their `slack_user_id` to a `primary_email` in the MongoDB `User` collection.

**FR-001: Avatar Asset Upload via Slack Modal**
*   **Priority:** SHALL
*   **Description:** The system shall allow authenticated users to upload their training assets and define media channels directly within Slack.
*   **Acceptance Criteria:**
    *   **Given** a user's identity is linked,
    *   **When** they continue the `/setup-avatar` flow,
    *   **Then** the system shall present a Slack Modal allowing them to upload 5-10 facial photos, upload a 30-second audio clip, and define the avatar's target "Channel" (e.g., "Professional LP", "Casual TikTok").

**FR-002: Avatar Asset Validation & Provisioning**
*   **Priority:** SHALL
*   **Description:** The system shall validate user-uploaded images via a local Clip+ViT vision model before triggering third-party training.
*   **Acceptance Criteria:**
    *   **Given** a user has submitted photos via the Slack Modal,
    *   **When** the images pass the local Clip+ViT lighting/occlusion checks,
    *   **Then** the system shall securely store assets in S3, dispatch training API calls to HeyGen and ElevenLabs, and set the profile status to `Training`.

**2. Data Ingestion & Event Routing**

**FR-003: Slack Webhook Event Ingestion**
*   **Priority:** SHALL
*   **Description:** The system shall expose an endpoint to securely ingest Slack events (text addresses or Google Maps pins) and immediately acknowledge the request to bypass Slack's 3-second timeout.
*   **Acceptance Criteria:**
    *   **Given** a user drops an address or pin into a designated Slack channel,
    *   **When** the FastAPI backend receives the event,
    *   **Then** it shall validate the HMAC-SHA256 signature, immediately return a `200 OK`, and push a `request.feasibility.new` event to RabbitMQ.
*   **API Endpoint:**
    *   `POST /api/v1/webhooks/slack/events`
    *   Request Schema: Standard Slack Events API JSON.
    *   Response Schema: `200 OK` (Empty body).

**FR-004: Geographic Parsing**
*   **Priority:** SHALL
*   **Description:** The system shall parse unstructured Slack text/links into standardized geographic coordinates and HCMC Ward/District identifiers using Gemini.
*   **Acceptance Criteria:**
    *   **Given** a RabbitMQ worker consumes a new feasibility request,
    *   **When** the payload is processed,
    *   **Then** the system shall extract standardized GPS coordinates and populate the `FeasibilityRequest` document in MongoDB.

**3. Multi-Agent Feasibility Engine**

**FR-005: Zoning & Compliance RAG Execution**
*   **Priority:** SHALL
*   **Description:** The Zoning Agent shall query a local Obsidian RAG database to determine Floor Area Ratio (FAR) and "Pink Book" zoning status for the provided coordinates.
*   **Acceptance Criteria:**
    *   **Given** parsed HCMC coordinates,
    *   **When** the Zoning Agent completes its query,
    *   **Then** it must output FAR, Max Floors, and Land-Use Status, assigning an explicit confidence score (0.0 to 1.0) to every extracted metric.

**FR-006: Financial Analysis & Market Context Execution**
*   **Priority:** SHALL
*   **Description:** The Financial Agent shall calculate ROI and Cap Rates. Concurrently, the Market Agent shall use Gemini for keyword trend extraction and `yt-dlp` to download top-performing local real estate videos for visual style analysis.
*   **Acceptance Criteria:**
    *   **Given** valid FAR data from the Zoning Agent,
    *   **When** the Market Agent executes,
    *   **Then** it shall query Gemini for trends, use `yt-dlp` to download 1-2 reference videos, pass them to the Vision Agent for B-roll/style cue extraction, and store the combined `analysis_results` in MongoDB.

**FR-007: Confidence Heatmap & Draft Generation**
*   **Priority:** SHALL
*   **Description:** The Orchestrator Agent shall synthesize agent outputs into a Dual-Mode text draft (LP Pitch & Social Hook) and inject a visual confidence heatmap.
*   **Acceptance Criteria:**
    *   **Given** the multi-agent analysis is complete,
    *   **When** the Orchestrator generates the text summary,
    *   **Then** it shall prepend every zoning/financial metric with 🔴 (Score < 0.80), 🟡 (0.80 - 0.89), or 🟢 (Score >= 0.90) and mandate manual review for red items.

**4. Interactive Workflow & Syndication**

**FR-008: Stage 1 Delivery & Tone Iteration**
*   **Priority:** SHALL
*   **Description:** The system shall deliver the initial draft back to the user's Slack thread as an interactive Block Kit message, allowing for script iteration and immediate text export.
*   **Acceptance Criteria:**
    *   **Given** the draft script is generated,
    *   **When** the Slack bot posts the message,
    *   **Then** the message must include interactive buttons: `[Make it Punchier]`, `[Make it Conservative]`, `[Export Summary]`, and `[Generate Video]`.
*   **API Endpoint:**
    *   `POST /api/v1/webhooks/slack/interactions`

**FR-009: Instant Deal Summary Export**
*   **Priority:** SHALL
*   **Description:** The system shall allow users to bypass the video rendering wait time by generating a clean, formatting-optimized text block or PDF of the feasibility summary.
*   **Acceptance Criteria:**
    *   **Given** a user clicks the `[Export Summary]` Slack button,
    *   **When** the action is received,
    *   **Then** the system shall instantly reply in the thread with a formatted text block optimized for copying into WhatsApp/Zalo, allowing immediate capital syndication.

**FR-010: Daily Credit Throttling**
*   **Priority:** SHALL
*   **Description:** The system shall verify the user's available daily API credit balance before allowing the transition to Stage 2 (Video Render).
*   **Acceptance Criteria:**
    *   **Given** a user clicks the `[Generate Video]` button,
    *   **When** the user has exhausted their daily generation credits,
    *   **Then** the system shall reject the request, log the attempt, and post a localized error message in the Slack thread.

**5. Video Content Orchestration & Final Delivery**

**FR-011: Asynchronous Media Rendering**
*   **Priority:** SHALL
*   **Description:** Upon user approval, the Orchestrator shall dispatch the approved script to ElevenLabs (audio) and HeyGen (video) based on the user's pre-configured channel avatar settings.
*   **Acceptance Criteria:**
    *   **Given** a user approves the script via `[Generate Video]`,
    *   **When** the credit check passes,
    *   **Then** the system shall transition the job status to `Rendering` and dispatch external API calls.

**FR-012: Secure Webhook Callback & Social-Ready Delivery**
*   **Priority:** SHALL
*   **Description:** The system shall process HeyGen video completion webhooks, store the asset, and deliver a "Social-Ready" message block to the user.
*   **Acceptance Criteria:**
    *   **Given** a job is in `Rendering` status,
    *   **When** the FastAPI backend receives a successful payload from HeyGen,
    *   **Then** it shall generate a 1-hour S3 Signed URL and post it to Slack alongside the final AI-generated script hook, completely formatted for one-tap copying to TikTok/Zalo.
*   **API Endpoint:**
    *   `POST /api/v1/webhooks/heygen/callback`
    *   Request Schema: `{"job_id": "string", "status": "string", "video_url": "string"}`

| Edge Case | Scenario | Expected System Behaviour |
| :--- | :--- | :--- |
| **Empty or Invalid Input** | User types "Check this out" or drops a pin outside of Ho Chi Minh City (e.g., Hanoi). | The Gemini Geo-Parser must detect the invalid/out-of-bounds input. The system shall immediately return a Slack message: *"I need a valid HCMC address or Google Maps pin to run feasibility."* The RabbitMQ event is not created. |
| **Imprecise Map Pin** | User drops a pin in a vaguely defined area (e.g., the middle of the Saigon River) where 1/500 zoning data does not apply. | The Zoning Agent must detect the lack of valid parcel data. It shall return a `REVIEW` status. The Orchestrator will draft a message stating: *"The coordinates provided do not map to a zoned parcel. Please adjust the pin closer to the street line."* |
| **Rapid Fire / Spam Inputs** | User drops 10 pins into the Slack channel within 5 seconds. | The database must deduplicate requests using the `slack_thread_ts` index. Subsequent requests within a 5-second window from the same user are ignored to prevent API/credit drain. |

**2. Edge Cases (AI & Agent Execution)**

| Edge Case | Scenario | Expected System Behaviour |
| :--- | :--- | :--- |
| **Zero-Silent-Failure Override** | The Zoning Agent confidently pulls FAR limits, but the Financial Modeler calculates a Cap Rate with < 0.80 confidence due to a lack of recent comps. | The Orchestrator Agent must detect the low score. It shall override the `GO` status to `REVIEW`, prepend the Cap Rate metric with a 🔴 emoji, and explicitly state in the script: *"Financial comps are sparse here; treat this yield projection as highly speculative."* |
| **Context Window Exceeded** | The user replies to the draft with an excessively long block of text (e.g., pasting a 10-page municipal document). | The Slack Listener must truncate the input to the first 1,000 tokens before passing it to the Orchestrator for tone iteration, appending a warning to the user that the input was truncated. |
| **Over-Scripting** | The Orchestrator Agent generates a text script that will take 90 seconds to read, exceeding the 15-60s video limitation. | The Output Parser must count the words (assuming ~150 words/minute). If > 150 words, it must force a single internal retry to the Orchestrator with a strict truncation prompt before presenting the draft to the user. |

**3. Edge Cases (External Integrations & APIs)**

| Edge Case | Scenario | Expected System Behaviour |

| **yt-dlp Scraping Blocked** | The Market Agent attempts to download a trending TikTok video for visual context, but the platform temporarily blocks the IP. | The Market Agent must gracefully catch the exception, skip the visual context injection, and proceed with the text-only Gemini trend data. The script generation must not fail. |

**4. Edge Cases (State & Network)**

| Edge Case | Scenario | Expected System Behaviour |
| :--- | :--- | :--- |
| **Slack Webhook Timeout** | The FastAPI backend takes 3.1 seconds to process the initial request, causing Slack to drop the connection and retry. | The FastAPI ingress must strictly enforce "Ack First, Process Later." ## Error Handling

**1. Error Taxonomy & System Degradation Strategy**

The system follows a strict "Zero Silent Failures" principle. Because inaccurate real estate underwriting carries high financial risk, the system favors explicit warnings and graceful degradation over attempting to proceed with uncertain data.

| Error Category | HTTP / System Code | Description | Global Mitigation Strategy |
| :--- | :--- | :--- | :--- |
| **Ingestion Errors** | `400 Bad Request` | Malformed Slack payloads, empty addresses, or bad HMAC signatures. | Immediate rejection. No DB/Queue entries created. |
| **Agentic Errors** | `AI_CONF_LOW` | LLM hallucinations or RAG data scarcity (Confidence $< 0.80$). | Force `REVIEW` status; degrade UI to include visual 🔴 warnings. |
| **External API Errors** | `502 / 504 / T/O` | ElevenLabs or HeyGen is down, rate-limited, or exceeds the 5-minute timeout SLA. | Circuit Breaker pattern. Auto-degrade to Text-Only delivery via Slack. |
| **State Errors** | `429 Too Many Req` | User attempts to generate video without sufficient daily credits. | Optimistic locking on MongoDB `$inc`; hard block with actionable UI message. |

**2. User Input & Validation Errors**

*   **Invalid Geographic Parsing:**
    *   **Failure Mode:** The Gemini Geo-Parser cannot extract a valid Ho Chi Minh City address or coordinates from the user's Slack message.
    *   **System Action:** The pipeline is immediately halted. The webhook acknowledges the event to Slack, but the system does not publish to RabbitMQ.
    *   **User Experience:** The bot replies in the thread: *"I couldn't pinpoint that location in HCMC. Please provide a more specific address or drop a Google Maps pin."*
*   **Media Upload Rejection (Clip+ViT):**
    *   **Failure Mode:** The user uploads photos during `/setup-avatar` that fail the local Vision validation (e.g., blurry, poor lighting, multiple faces).
    *   **System Action:** The system deletes the temporary files and halts the third-party training API dispatch.
    *   **User Experience:** The bot sends a localized DM: *"Your avatar photos couldn't be processed. Ensure there is only one face in the frame and the lighting is clear. Please try uploading again."*

**3. AI & Agentic Reasoning Errors (Zero Silent Failures)**

*   **Zoning RAG Data Scarcity:**
    *   **Failure Mode:** The Zoning Agent cannot find explicit 1/500 plans for the provided coordinates in the Obsidian database, resulting in a confidence score $< 0.80$.
    *   **System Action:** The Output Validator forcibly overrides the overall feasibility status to `REVIEW`. It prepends the extracted metrics with a red 🔴 emoji.
    *   **User Experience:** The Slack draft displays: *"🔴 **WARNING:** Zoning data for this ward is incomplete. The FAR of 4.5 is an estimation based on adjacent parcels. Manual 'Pink Book' verification is strictly required."*
*   **Prompt/Context Truncation:**
    *   **Failure Mode:** The user replies to a draft script with an excessively long prompt (e.g., pasting a 10-page municipal code), exceeding the Gemini context window limits for rapid iteration.
    *   **System Action:** The Slack Listener truncates the input to the first 1,000 tokens before passing it to the Orchestrator.
    *   **User Experience:** The system processes the request but appends an informational warning: *"Note: Your prompt was very long, so I only used the first part to adjust the script tone."*

**4. External API & Infrastructure Errors**

*   **HeyGen Render Timeout (The 5-Minute SLA):**
    *   **Failure Mode:** The HeyGen API accepts the video generation job but fails to return a successful webhook within the 5-minute SLA.
    *   **System Action:** A background Cron sweeper detects `VideoGenerationJob.updated_at > 5 mins` while still in `Rendering` state. It terminates waiting for the webhook and forces a state transition to `Delivered (Fallback)`.
    *   **User Experience:** The Slack bot pushes the text-only summary into the thread with an alert: *"Video rendering is experiencing delays at the provider. Here is your text-based summary so you can proceed with your underwriting immediately."*
*   **ElevenLabs Voice Generation Failure:**
    *   **Failure Mode:** ElevenLabs returns a 4xx/5xx error (e.g., due to API downtime or prompt rejection).
    *   **System Action:** The Orchestrator implements a 3-attempt retry policy with exponential backoff (2s, 4s, 8s). If the 3rd attempt fails, it triggers the text-only fallback.
    *   **User Experience:** The system skips video generation and alerts the user: *"The voice generation service is temporarily down. Your text summary has been delivered instead."*
*   **Slack Webhook Ack Timeout Avoidance:**
    *   **Failure Mode:** The FastAPI backend takes $> 3.0$ seconds to process an incoming Slack event, causing Slack to assume failure and retry the payload.
    *   **System Action:** The `POST /api/v1/webhooks/slack/events` route acts strictly as a proxy. It validates the HMAC signature, pushes the raw payload to RabbitMQ, and returns `200 OK` in $< 500$ms.
    *   **User Experience:** The user never experiences duplicate bot replies or timeout errors; the processing is guaranteed to be asynchronous.

**5. State & Security Errors**

*   **Insufficient Daily Credits:**
    *   **Failure Mode:** The user clicks `[Generate Video]` but has exhausted their daily API credit limit (enforcing the "low cost" unit economics guardrail).
    *   **System Action:** The MongoDB `$inc` operation fails the optimistic lock check. The system halts the API dispatch to ElevenLabs/HeyGen.
    *   **User Experience:** The bot replies in the thread: *"You have used all your video generation credits for today. You can continue to generate unlimited text drafts, or upgrade your tier to render more videos."*

## Success Metrics

**1. Primary North Star Metric (Capital Velocity)**

*   **Metric:** Time-to-Feasibility & Syndication (TTFS)
*   **Definition:** The total elapsed time from the user dropping a pin/address in Slack to the successful generation and delivery of the final video URL.
*   **Baseline:** 21 days (industry average for manual 1/500 zoning validation, financial underwriting, and pitch creation).
*   **Target:** $< 8$ minutes per request.
*   **Rationale:** This metric directly measures the core value proposition: collapsing a multi-week underwriting and syndication process into a single, on-site mobile interaction.

**2. System SLA & Performance Metrics**

*   **Metric:** Slack Acknowledgment Latency
*   **Definition:** Time elapsed between receiving the Slack webhook and returning a 200 OK response.
*   **Target:** 100% of requests acknowledged in $< 3.0$ seconds (Slack's mandatory timeout window).
*   **Instrumentation:** Datadog/APM tracing on the FastAPI `POST /api/v1/webhooks/slack/events` route.

*   **Metric:** Text-Draft Generation Latency (p95)
*   **Definition:** The time from webhook ingestion to the delivery of the Stage 1 text-based feasibility draft in the Slack thread.
*   **Target:** 95th percentile (p95) completed in $< 45$ seconds.

*   **Metric:** Asynchronous Video Render Delivery (p95)
*   **Definition:** The time from the user clicking `[Generate Video]` to the successful delivery of the HeyGen S3 Signed URL.
*   **Target:** 95th percentile (p95) completed in $< 5$ minutes.

**3. AI Quality & Trust Metrics (Risk Mitigation)**

*   **Metric:** Zero Silent Failures Adherence
*   **Definition:** The percentage of AI-generated data points with a confidence score $< 0.80$ that successfully trigger a forced `REVIEW` status and a visible 🔴 warning in the Slack draft.
*   **Target:** 100% adherence (Critical business constraint).
*   **Instrumentation:** Unit tests simulating $< 0.80$ confidence JSON payloads; real-time monitoring comparing raw LLM output logs against the final delivered Slack Block Kit payload.

*   **Metric:** Agent Confidence Frequency (RAG Quality Indicator)
*   **Definition:** The average confidence score generated by the Zoning Agent across all requests within specific HCMC wards/districts.
*   **Target:** Maintain a moving average of $> 0.85$ confidence globally.
*   **Rationale:** Identifying specific districts that consistently trigger low confidence scores ($< 0.70$) serves as a leading indicator to tell the data engineering team exactly which local municipal 1/500 plans need to be manually updated in the Obsidian knowledge base.

**4. Value Realization & Engagement Metrics**

*   **Metric:** Script-to-Video Conversion Rate
*   **Definition:** The percentage of generated text feasibility drafts (Stage 1) that result in the user clicking `[Generate Video]` (Stage 2).
*   **Target:** $> 40\%$ conversion rate.
*   **Rationale:** If users are requesting the initial text analysis but abandoning the workflow before generating the video, the product is succeeding as an analytics tool but failing as a *syndication* engine.

*   **Metric:** Instant Deal Summary Export Rate
*   **Definition:** The percentage of users clicking the `[Export Summary]` button to bypass the video render and immediately share the text-based data.
*   **Target:** Establish baseline in Month 1.
*   **Rationale:** Measures the immediate "capital velocity" demand. High usage here indicates that raw speed to share data outweighs the polish of the avatar video in certain deal scenarios.

**5. Unit Economics & Cost Metrics**

*   **Metric:** Average API Cost per Feasibility Run
*   **Definition:** The blended cost of Gemini (reasoning), ElevenLabs (audio), and HeyGen (video) API calls per completed workflow.
*   **Target:** Keep average cost strictly below the revenue threshold defined by the daily user subscription tier (enforced via the Daily Credit limit).
*   **Instrumentation:** Aggregated billing dashboards correlating MongoDB `VideoGenerationJob` completions with external API billing logs.
