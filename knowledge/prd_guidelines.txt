================================================================================
PRD WRITING GUIDELINES — CrewAI Product Feature Planner
================================================================================

This document defines the standards, expectations, and quality criteria for
every section of a Product Requirements Document generated by this system.
Agents must follow these guidelines when drafting and critiquing PRD sections.

────────────────────────────────────────────────────────────────────────────────
1. PRD SECTION ORDER AND EXPECTATIONS
────────────────────────────────────────────────────────────────────────────────

Every PRD contains exactly 10 sections, generated in the following order:

Section 1: Executive Summary
  Purpose: High-level overview of the product feature.
  Must include:
    • Clear problem statement (1–2 sentences)
    • Proposed solution overview
    • Target audience identification
    • Key business value / ROI justification
    • High-level timeline or phasing indication
  Quality bar: Concise, compelling, self-contained — a reader should
  understand the entire feature from this section alone.

Section 2: Problem Statement
  Purpose: Articulate the pain point this feature addresses.
  Must include:
    • Current state description (what exists today)
    • Pain points with evidence (user research, data, support tickets)
    • Impact quantification (lost revenue, wasted time, churn risk)
    • Why now? — urgency or market timing justification
  Quality bar: Data-driven where possible. Avoid vague language like
  "users are frustrated" without specifics.

Section 3: User Personas
  Purpose: Define who will use this feature and how.
  Must include:
    • 2–4 distinct personas with names, roles, and demographics
    • Pain points specific to each persona
    • Goals and desired outcomes for each persona
    • Usage frequency and context (daily power user vs. occasional)
  Quality bar: Personas must be specific enough to drive design decisions.

Section 4: Functional Requirements
  Purpose: Exhaustive list of what the system must do.
  Must include:
    • Numbered requirements (FR-001, FR-002, ...)
    • Each requirement: SHALL/SHOULD/MAY priority level
    • Acceptance criteria in Given/When/Then format
    • Input validation rules and expected outputs
    • API endpoints with method, path, request/response schemas
  Quality bar: A developer should be able to implement any single
  requirement without asking clarifying questions.

Section 5: Non-Functional Requirements
  Purpose: Quality attributes and system constraints.
  Must include:
    • Performance: latency targets (p50, p95, p99), throughput (RPS)
    • Scalability: concurrent users, data volume projections
    • Availability: uptime SLA (e.g. 99.9%), RTO, RPO
    • Security: authentication, authorisation, data encryption
    • Compliance: GDPR, SOC 2, HIPAA if applicable
    • Accessibility: WCAG level target
  Quality bar: Every NFR must have a measurable, testable target.

Section 6: Edge Cases
  Purpose: Scenarios outside the happy path.
  Must include:
    • Boundary conditions (empty inputs, max-length values)
    • Concurrent access scenarios (race conditions, optimistic locking)
    • Network failure handling (offline mode, retry strategies)
    • Data inconsistency scenarios
    • User behaviour edge cases (rapid clicking, back-button, tab switching)
  Quality bar: Each edge case must specify the expected system behaviour.

Section 7: Error Handling
  Purpose: How the system responds to failures.
  Must include:
    • Error taxonomy (categories with HTTP status codes or error codes)
    • User-facing error messages (clear, actionable, non-technical)
    • System-level error handling (retry, circuit breaker, dead-letter queue)
    • Logging and alerting requirements per error category
    • Graceful degradation strategy
  Quality bar: No silent failures. Every error must be logged, surfaced,
  or handled with a defined fallback.

Section 8: Success Metrics
  Purpose: How to measure if the feature succeeds.
  Must include:
    • Primary KPI with baseline and target (e.g. "reduce onboarding
      time from 5 min to 2 min within 90 days of launch")
    • Secondary metrics (adoption rate, NPS delta, support ticket volume)
    • Instrumentation requirements (events, properties, dashboards)
    • Experiment design (A/B test plan if applicable)
    • Leading vs. lagging indicators
  Quality bar: Metrics must be specific, measurable, time-bound, and
  directly attributable to this feature.

Section 9: Dependencies
  Purpose: External factors that could block or affect delivery.
  Must include:
    • Technical dependencies (services, APIs, infrastructure)
    • Team dependencies (design, backend, data engineering, QA)
    • Third-party dependencies (vendors, licenses, contracts)
    • Risk assessment for each dependency (likelihood × impact)
    • Mitigation strategies for high-risk dependencies
  Quality bar: Every dependency has an owner and a mitigation plan.

Section 10: Assumptions
  Purpose: Statements accepted as true without proof.
  Must include:
    • Business assumptions (market size, user willingness to pay)
    • Technical assumptions (available APIs, infrastructure capacity)
    • Operational assumptions (team capacity, timeline feasibility)
    • Validation plan for critical assumptions
  Quality bar: Each assumption must be explicit and falsifiable.

────────────────────────────────────────────────────────────────────────────────
2. QUALITY CRITERIA
────────────────────────────────────────────────────────────────────────────────

Every section is scored 1–10 on six dimensions:

  1. Completeness — Does it address all required elements?
  2. Specificity — Are requirements concrete and testable?
  3. Consistency — Does it align with the executive summary and other sections?
  4. Clarity — Is the language unambiguous and well-organised?
  5. Actionability — Can a developer act on it without questions?
  6. No Duplication — Does it avoid repeating content from other sections?

A section is approved ("SECTION_READY") when ALL six scores reach ≥ 8.
A section needs refinement ("NEEDS_REFINEMENT") when any score is < 8.

────────────────────────────────────────────────────────────────────────────────
3. WRITING STYLE
────────────────────────────────────────────────────────────────────────────────

• Use active voice and present tense ("The system validates..." not
  "The system should validate...").
• Use consistent terminology — define terms on first use.
• Use numbered lists for requirements, bullet lists for descriptions.
• Use tables for structured data (API endpoints, metrics, personas).
• Use headings and sub-headings to organise long sections.
• Avoid implementation details unless they constrain the solution space.
• Cross-reference other sections by name when relevant.
• Include concrete examples where they clarify the requirement.

────────────────────────────────────────────────────────────────────────────────
4. ITERATION PROTOCOL
────────────────────────────────────────────────────────────────────────────────

Each section goes through a minimum of 2 and maximum of 10 draft-critique
cycles.  On each cycle:

  1. DRAFT — Write or revise the section content addressing all critique points.
  2. CRITIQUE — Score the draft on the 6 quality dimensions (1–10 each).
  3. DECIDE — If all scores ≥ 8 → approve. Otherwise, provide specific
     feedback and iterate.

Degenerate output: If a draft is excessively short, off-topic, or contains
template placeholders, the content is wiped and the section restarts from
scratch.  This prevents propagation of low-quality output.

────────────────────────────────────────────────────────────────────────────────
5. ANTI-PATTERNS TO AVOID
────────────────────────────────────────────────────────────────────────────────

✗ Vague requirements ("The system should be fast")
✗ Missing acceptance criteria
✗ Duplicating content across sections
✗ Requirements without priority levels
✗ Metrics without baselines or targets
✗ Edge cases without specified behaviour
✗ Dependencies without owners or mitigations
✗ Assumptions presented as facts
✗ Implementation-specific language in requirements (unless intentional)
✗ Placeholder text or TODO markers in final output
