================================================================================
PROJECT ARCHITECTURE — CrewAI Product Feature Planner
================================================================================

This document describes the architecture, structure, and conventions of the
CrewAI Product Feature Planner system.  Agents should reference this when
making decisions about PRD content, technical feasibility, and output format.

────────────────────────────────────────────────────────────────────────────────
1. SYSTEM OVERVIEW
────────────────────────────────────────────────────────────────────────────────

The CrewAI Product Feature Planner is a multi-agent system that transforms
raw product ideas into comprehensive, implementation-ready Product Requirements
Documents (PRDs).  It runs as a FastAPI server with a REST API and uses
MongoDB for persistence.

Tech stack:
  • Python 3.11
  • CrewAI 1.9.3 (multi-agent orchestration framework)
  • Google Gemini LLM (default model: gemini-3-flash-preview)
  • OpenAI o3 (fallback / secondary PM agent)
  • MongoDB (working ideas, finalized ideas, crew jobs)
  • FastAPI (REST API + Slack webhooks)
  • Slack (Bot integration: Events API, Interactions API, OAuth v2,
    Block Kit interactive messages, token rotation)
  • Atlassian Confluence + Jira (post-completion publishing)

Project configuration notes:
  • The Slack setup wizard collects only Confluence space key and Jira project key.
  • Confluence parent page ID is optional and may be provided via project config
    or the CONFLUENCE_PARENT_ID env var. If unset, PRDs publish at the space root.
  • LLM prompts should not ask users to provide a parent page ID.

3. PRD GENERATION PIPELINE
────────────────────────────────────────────────────────────────────────────────

The PRD generation pipeline has five sequential phases:

Phase 0 — Pre-completion Pipeline (AgentOrchestrator):
  Stage 1: Idea Refinement (Gemini) — 3–10 iterative cycles
    • Evaluates: audience clarity, problem definition, solution specificity,
      competitive context, success criteria (scored 1–5)
    • Continues until all scores ≥ 3 ("IDEA_READY")
  Stage 2: Requirements Breakdown (Gemini) — iterative cycles
    • Decomposes idea into: entities, state machines, API contracts,
      acceptance criteria
    • Evaluates: feature completeness, entity granularity, state machine
      rigour, AI augmentation depth, API contract precision,
      acceptance criteria coverage (scored 1–5, all must ≥ 4)

Phase 1 — Executive Summary Iteration:
  • Draft → Critique → Refine loop (min 2, max 10 cycles)
  • Uses multi-agent setup: Gemini PM + OpenAI PM
  • Critique scores: problem clarity, personas, functional reqs,
    non-functional reqs, edge cases, analytics, dependencies (rated 1–10)
  • Approved when all scores ≥ 8 ("READY_FOR_DEV")

Phase 2 — Section-by-Section Generation:
  • 9 remaining sections processed sequentially
  • Each section: Draft → Critique → Refine (min 2, max 10 cycles)
  • Critique scores: completeness, specificity, consistency, clarity,
    actionability, no-duplication (rated 1–10, all must ≥ 8)
  • Approved when "SECTION_READY"

Phase 3 — Finalization:
  • Assemble all 10 sections into markdown
  • Write PRD file to output/prds/
  • Mark run as completed in MongoDB

Phase 4 — Post-completion Pipeline (AgentOrchestrator):
  Stage 1: Confluence Publish — push final PRD to Confluence space
  Stage 2: Jira Ticketing — create Epic + Stories from functional requirements

3a. SLACK INTERACTIVE FLOW
────────────────────────────────────────────────────────────────────────────────

The system supports a full Slack-native interactive flow that mirrors the CLI
experience inside Slack using Block Kit buttons and threaded conversations.

Entry points:
  • POST /slack/kickoff with interactive=true (API call)
  • @mention the bot with keywords: "interactive", "step by step", "guided"

Interactive flow phases:

  1. Refinement Mode Choice — Block Kit buttons ask the user:
     • "Agent" — Gemini Idea Refiner runs automatically (3–10 cycles)
     • "Manual" — user types refinements in the thread, clicks Approve
       when satisfied (thread-based loop with approve/cancel buttons)

  2. Idea Approval — Block Kit buttons show refined idea:
     • "Approve" — proceed to requirements breakdown
     • "Cancel" — stop the flow, post cancellation notice

  3. Requirements Approval — Block Kit buttons show structured requirements:
     • "Approve" — proceed to auto-generate PRD sections
     • "Cancel" — stop the flow

  4. Auto-Generation — phases 1–4 of the standard pipeline run with
     auto_approve=true, posting results back to the Slack channel/thread.

Block Kit action IDs:
  • refinement_agent, refinement_manual — refinement mode choice
  • idea_approve, idea_cancel — idea approval gate
  • requirements_approve, requirements_cancel — requirements gate
  • flow_cancel — cancel at any point

State management:
  • _interactive_runs dict tracks per-run state with threading.Event
    for blocking until user makes a decision
  • 30-minute TTL for stale entries; 10-minute timeout per decision
  • Resolved by interactions_router → resolve_interaction() which
    signals the waiting thread

Slack webhooks:
  • POST /slack/events — receives Events API callbacks (url_verification,
    member_joined_channel, app_mention, threaded message)
  • POST /slack/interactions — receives Block Kit button clicks
    (block_actions, view_submission)
  • GET /slack/oauth/callback — OAuth v2 install/reinstall callback
  • All requests verified via HMAC-SHA256 signing secret

4. AGENT ROLES
────────────────────────────────────────────────────────────────────────────────

PRD Planning Agents:
  • Idea Refiner (Gemini, no tools)
    Purpose: Enrich raw ideas with domain expertise and stakeholder perspective.
    Iterates 3–10 cycles until all quality scores >= 3 (IDEA_READY).

  • Product Manager — Gemini variant (6 tools: search, scrape, file I/O)
    Purpose: Draft and critique PRD sections with deep industry knowledge.

  • Product Manager — OpenAI variant (same 6 tools)
    Purpose: Secondary PM for multi-agent diversity in critique cycles.

  • Requirements Breakdown Agent (Gemini, no tools)
    Purpose: Decompose refined ideas into entities, state machines,
    API contracts, and acceptance criteria.

  • CEO Reviewer (Gemini, no tools)
    Purpose: Transform executive summary into 10-star Executive Product
    Summary with visionary scope and delight opportunities.

  • Engineering Manager (Gemini, no tools)
    Purpose: Produce comprehensive engineering plans with architecture,
    phased implementation, ASCII diagrams, and test strategy.

UX Design Agents:
  • UX Designer (Gemini, 2 tools: search, scrape)
    Purpose: Generate initial design specification with design system,
    user flows, components, and CSS tokens.

  • Design Partner (Gemini, no tools)
    Purpose: Collaborate with UX Designer on aesthetic direction, design
    system research, and AI slop avoidance.

  • Senior Designer (Gemini, no tools)
    Purpose: 7-pass review of design specification to produce final,
    production-ready design spec.

Quality Review Agents:
  • Staff Engineer (Gemini, 1 tool: Jira)
    Purpose: Paranoid structural audit — N+1 queries, race conditions,
    trust boundaries, broken invariants, bad retry logic.

  • QA Lead (Gemini, 1 tool: Jira)
    Purpose: Test methodology audit — acceptance criteria coverage,
    test gaps, regression risk, test data requirements.

  • QA Engineer (Gemini, 1 tool: Jira)
    Purpose: Edge case, security, and rendering/behaviour test review.
    Creates counter-tickets for every implementation sub-task.

Delivery Agents:
  • Orchestrator Agent (Gemini, 2 tools: Confluence + Jira)
    Purpose: Publish completed PRDs and create tracking tickets.

  • Architect Tech Lead (Gemini, 1 tool: Jira)
    Purpose: Decompose User Stories into sub-tasks with full
    implementation specs for AI coding agents.

  • Product Manager Jira (Gemini, 1 tool: Jira)
    Purpose: Create agent-ready Jira Epics and Stories with acceptance
    criteria scoped by discipline.

  • Delivery Manager (Gemini, 2 tools: Confluence + Jira)
    Purpose: Autonomous startup delivery scanning for unpublished PRDs.

Coordination Agents:
  • Engagement Manager (Gemini, 5 tools: Slack + file I/O)
    Purpose: Orchestrate full idea-to-PRD lifecycle with heartbeat
    updates, user steering, and session isolation.

  • Idea Agent (Gemini, 2 tools: file I/O)
    Purpose: Real-time iteration advisor during active PRD flows.
    Answers questions about current state and steers iterations.

Utility Agents (outside PRD flow):
  • Release Engineer
    Purpose: Release hygiene — sync, test, push, create PR.

  • Retro Manager
    Purpose: Retrospective analysis from commit history with metrics.

5. STATE MANAGEMENT
────────────────────────────────────────────────────────────────────────────────

PRDState tracks the full lifecycle of a PRD generation run:
  • run_id — unique 12-char hex identifier
  • idea — the feature idea text (enriched after refinement)
  • original_idea — snapshot before refinement
  • draft — PRDDraft containing 10 PRDSection objects
  • executive_summary — iterative draft with version history
  • requirements_breakdown — structured requirements text
  • status — "new" → "inprogress" → "completed"
  • confluence_url — URL of published Confluence page
  • jira_output — summary of created Jira tickets

Progress is saved to MongoDB on every iteration, enabling pause/resume.
On server restart, incomplete runs are detected and can be resumed.

5a. SLACK INTERACTIVE STATE
────────────────────────────────────────────────────────────────────────────────

Slack interactive flows maintain additional in-memory state:

  _interactive_runs[run_id] → {
    channel, thread_ts, user, idea,     # Slack context
    created_at,                          # TTL tracking (30 min)
    pending_action,                      # Current wait type
    event (threading.Event),             # Signals user decision
    decision,                            # Action ID from button click
    cancelled,                           # Boolean cancellation flag
  }

Thread conversations:
  • Events router maintains _thread_conversations per (channel, thread_ts)
  • 10-minute TTL, 20-message rolling window
  • Deduplication by event_id with 5-minute TTL

Manual refinement:
  • _manual_refinement_text[run_id] stores latest thread reply
  • Thread messages route to submit_manual_refinement() when
    pending_action == "manual_refinement"

9. PUBLISHING AUTOMATION
────────────────────────────────────────────────────────────────────────────────

Three complementary mechanisms ensure PRDs are reliably delivered to
Confluence and Jira:

a) API + Slack Intent (manual trigger):
   Users can say "publish all" or "check publishing status" in Slack,
   or call the /publishing/* REST endpoints directly.  Gemini classifies
   these as "publish" or "check_publish" intents.

b) File Watcher (auto-trigger on file create):
   A polling-based background thread watches output/prds/ for new .md
   files.  When a file stabilises (size unchanged across two polls),
   it auto-publishes to Confluence.  Configurable via env vars:
     PUBLISH_WATCHER_ENABLED     — "0"/"false" to disable (default: on)
     PUBLISH_WATCHER_POLL_SECONDS — polling interval (default: 10)

c) Cron Scheduler (periodic scan for resilience):
   A background thread sweeps for incomplete deliveries on a timer,
   resuming any Confluence or Jira work that was interrupted by server
   restarts or transient failures.  Configurable via env vars:
     PUBLISH_SCHEDULER_ENABLED          — "0"/"false" to disable
     PUBLISH_SCAN_INTERVAL_SECONDS      — sweep interval (default: 300 = 5 min)

Both watcher and scheduler start automatically in the FastAPI lifespan
when Confluence credentials are configured and are cleanly stopped
on server shutdown.

10. AGENT INTERACTION TRACKING
────────────────────────────────────────────────────────────────────────────────

Every user↔agent interaction from Slack and CLI is logged to the MongoDB
`agentInteraction` collection for future LLM fine-tuning data.

Collection: agentInteraction
Module:     mongodb/agent_interactions/repository.py

Document schema:
  interaction_id          — UUID hex (primary key)
  source                  — "slack" | "slack_interactive" | "cli"
  user_message            — Original user input text
  intent                  — Classified intent (create_prd, help, greeting,
                            publish, check_publish, unknown, refinement_mode,
                            manual_refinement, manual_refinement_approve,
                            manual_refinement_edit, idea_approval,
                            requirements_approval)
  agent_response          — Agent reply text
  idea                    — Extracted product idea (if applicable)
  run_id                  — Associated PRD flow run ID (if applicable)
  channel                 — Slack channel ID (Slack sources only)
  thread_ts               — Slack thread timestamp (Slack sources only)
  user_id                 — Slack user ID or "cli_user"
  conversation_history    — Thread conversation context (Slack only)
  metadata                — Freeform dict (iteration counts, revised text, etc.)
  created_at              — UTC datetime

Tracking integration points:
  • apis/slack/events_router.py — _interpret_and_act() logs after each
    Gemini intent classification (6 intents)
  • apis/slack/interactive_handlers.py — resolve_interaction() logs button
    clicks; submit_manual_refinement() logs thread replies
  • components/cli.py — _track_cli() helper logs at every CLI decision
    point (idea input, refinement mode, manual edits, approvals)

All tracking calls are wrapped in try/except to ensure tracking failures
never disrupt the main user flow.

Repository functions:
  log_interaction(…)              — Insert a new interaction document
  get_interaction(id)             — Fetch single interaction by ID
  find_interactions_by_source(…)  — Filter by source, newest first
  find_interactions_by_intent(…)  — Filter by intent, newest first
  find_interactions(…)            — Flexible multi-filter query
  list_interactions(…)            — All recent interactions
